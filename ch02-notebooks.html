Chapter 2

Notebooks & Dev Patterns

dbutils, magic commands, %run, display(), widgets — the toolbox that makes Databricks notebooks more than just Jupyter.

The Databricks Notebook

Databricks notebooks look like Jupyter but are fundamentally different. They run on a remote cluster, support multiple languages in the same notebook, share state across cells (and across %run calls), and integrate tightly with the Databricks platform. A few important differences from local Jupyter:

  • State is cluster-state, not kernel-state. Variables persist across cell executions and across notebook sessions attached to the same cluster. To get a fresh slate: click the cluster name dropdown at the top right of the notebook and select Detach & Re-attach, or choose Clear state & run all from the Run menu.
  • Cell outputs are rich. DataFrames render as interactive tables, plots render inline, and you can build dashboards directly from cell outputs.
  • Notebooks are version-controlled. Each save creates a Git-compatible snapshot. You can connect notebooks directly to a Git repo.

Magic Commands

Magic commands switch the language or context for a single cell. They must be the first line of the cell:

MagicEffectCommon Use
%pythonRun cell as Python (default)PySpark, pandas, general logic
%sqlRun cell as SQLQuerying tables, DDL, quick data checks
%scalaRun cell as ScalaLibraries only available in Scala
%rRun cell as RStatistical analysis with R packages
%mdRender cell as MarkdownDocumentation, section headers
%shRun shell command on driverChecking disk space, installing packages, pinging endpoints
%pipInstall Python packagesAdding libraries without cluster restart (DBR 7.1+)
%runExecute another notebookCode reuse, shared utilities
%fsDBFS filesystem commandsls, cp, rm, mkdirs on DBFS
 
# %pip installs packages into the current notebook session
# Place at the top of the notebook before imports
# COMMAND ----------
# NOTE: "# COMMAND ----------" is the cell separator used when notebooks are
# exported as .py files. In the Databricks UI, each cell is a separate block.
%pip install great-expectations==0.18.0

# COMMAND ----------
# Now import normally
import great_expectations as gx
%pip Restarts the Python Kernel

%pip install triggers a Python kernel restart, which clears all variable state. Any variables defined in cells above will be lost. Always put %pip cells at the very top of your notebook, before any computation, and run top-to-bottom after adding a new dependency.

dbutils: The Databricks Swiss Army Knife

dbutils is a Databricks-specific utility module pre-loaded in every notebook. It has five main components:

dbutils.fs — Filesystem Operations

# List files
dbutils.fs.ls("s3://my-bucket/data/")
dbutils.fs.ls("dbfs:/tmp/")

# Copy
dbutils.fs.cp("dbfs:/tmp/source.csv", "dbfs:/tmp/dest.csv")

# Move (rename)
dbutils.fs.mv("dbfs:/tmp/old-name/", "dbfs:/tmp/new-name/", recurse=True)

# Remove
dbutils.fs.rm("dbfs:/tmp/stale-data/", recurse=True)

# Make directory
dbutils.fs.mkdirs("dbfs:/tmp/new-dir/")

# Read a text file as a string
content = dbutils.fs.head("dbfs:/tmp/config.json", maxBytes=10000)
print(content)
If You Know Basic PySpark

You might be used to Python's os module for filesystem operations. On Databricks, os.listdir() sees the driver's local filesystem — not DBFS or S3. Always use dbutils.fs for anything on DBFS or cloud storage, and os only for driver-local temp files.

dbutils.secrets — Secrets Management

Never hardcode credentials in notebooks. Use Databricks secrets scopes:

# Retrieve a secret (never prints the value — returns a redacted string in UI)
api_key = dbutils.secrets.get(scope="my-aws-scope", key="s3-access-key")

# List available secret scopes
dbutils.secrets.listScopes()

# List secrets in a scope (shows key names only, not values)
dbutils.secrets.list("my-aws-scope")

When you call dbutils.secrets.get(), the value is returned as a Python string but is automatically redacted if it appears in cell output or logs. The string itself can be used normally in your code.

dbutils.widgets — Parameterizing Notebooks

Widgets let you create notebook parameters that can be set from the UI or passed from a job. They're the Databricks equivalent of Jupyter ipywidgets, but they also integrate with Databricks Jobs:

# Define widgets (creates UI controls at top of notebook)
dbutils.widgets.text("environment", "dev", "Environment")
dbutils.widgets.dropdown("table_date", "2024-01-01",
    ["2024-01-01", "2024-01-08", "2024-01-15"], "Table Date")
dbutils.widgets.combobox("output_format", "delta",
    ["delta", "parquet", "csv"], "Output Format")
dbutils.widgets.multiselect("regions", "US",
    ["US", "EU", "APAC"], "Regions")

# Read widget values
env = dbutils.widgets.get("environment")
date = dbutils.widgets.get("table_date")
fmt = dbutils.widgets.get("output_format")
regions = dbutils.widgets.get("regions").split(",")  # multiselect returns CSV string

print(f"Running for env={env}, date={date}, regions={regions}")
Widgets in Production Jobs

When a notebook runs as part of a Databricks Job, widget values are supplied as job parameters. In the job definition, you map parameter names to widget names. The notebook code doesn't change — dbutils.widgets.get("environment") gets the value whether it comes from the UI or a job parameter. This is why widgets are the right way to parameterize production notebooks.

dbutils.notebook — Notebook Orchestration

# Run another notebook synchronously and get its return value
result = dbutils.notebook.run(
    "/path/to/child-notebook",
    timeout_seconds=3600,
    arguments={"environment": "prod", "table_date": "2024-01-01"}
)
print(result)  # the value passed to dbutils.notebook.exit() in the child

# Exit a notebook with a return value (used in child notebooks)
dbutils.notebook.exit("SUCCESS: 1,234,567 rows processed")

%run: Importing Utilities Across Notebooks

%run executes another notebook in the current notebook's context. All functions, classes, and variables defined in the target notebook become available in the calling notebook. Think of it like Python's exec(open("file.py").read()) — not an import, but a full execution in the same namespace:

# In a shared utility notebook at /Shared/utils/spark_helpers
def read_delta_table(catalog, schema, table):
    return spark.read.table(f"{catalog}.{schema}.{table}")

def write_delta_table(df, catalog, schema, table, mode="overwrite"):
    df.write.format("delta").mode(mode).saveAsTable(f"{catalog}.{schema}.{table}")

# In your main notebook:
# COMMAND ----------
%run /Shared/utils/spark_helpers

# COMMAND ----------
# These functions are now available
df = read_delta_table("my_catalog", "bronze", "raw_orders")
write_delta_table(cleaned_df, "my_catalog", "silver", "orders")
Databricks Repos and Python Modules

Databricks Repos lets you connect your workspace to a Git repository (GitHub, GitLab, Azure DevOps) and check out branches directly. Files in a Repo that end in .py are importable as regular Python modules with import or from ... import — the standard Python way. This is the preferred approach for serious production code. Navigate to the Repos section in the left sidebar to set up your first repo.

%run Gotchas

Re-execution cost: %run re-executes the entire notebook every time, including any expensive setup code. For large shared utilities, use Databricks Repos with proper Python modules (.py files) and import them instead. Variable collision: variables in the target notebook overwrite same-named variables in the caller. Name your utility variables carefully (underscore prefix or module-style namespacing).

display() vs show() vs toPandas()

This is one of the most common points of confusion for engineers coming from local PySpark:

Local PySpark
# .show() is the standard way to view data
df.show(20, truncate=False)

# Output: plain text table
# +----------+-----+
# |first_name|score|
# +----------+-----+
# |Alice     |   92|
# +----------+-----+

# .toPandas() for richer display in Jupyter
df.toPandas().head(20)
Databricks
# display() renders an interactive rich table
display(df)

# Supports sorting, filtering, pagination, charts
# Does NOT pull all data to driver — paginates intelligently

# .show() still works but produces plain text
df.show(20)

# .toPandas() still works — pulls ALL rows to driver
# Dangerous on large DataFrames — set row limit first
df.limit(10000).toPandas()

Always prefer display(df) for interactive exploration in Databricks. Use .show() when you need text output (e.g., in job logs). Use .toPandas() only when you explicitly need a pandas DataFrame, and always limit rows first.

Mixing SQL and Python

Spark SQL and the DataFrame API interoperate freely. You can switch between them at any point using spark.sql() and createOrReplaceTempView():

# Register a DataFrame as a temporary view
df.createOrReplaceTempView("orders_raw")

# Query it with SQL
result = spark.sql("""
    SELECT
        customer_id,
        COUNT(*) AS order_count,
        SUM(amount) AS total_spent
    FROM orders_raw
    WHERE status = 'completed'
    GROUP BY customer_id
    HAVING total_spent > 1000
""")

# Result is a DataFrame — continue with PySpark API
result.filter(F.col("order_count") > 5).display()

The %sql magic is syntactic sugar for spark.sql() — the result is displayed as a table and also accessible as _sqldf in the next Python cell. Note: _sqldf is overwritten every time a %sql cell runs, so assign it to a named variable immediately if you need to use it more than once.

-- %sql cell
SELECT customer_id, SUM(amount) as total
FROM orders_raw
GROUP BY customer_id
ORDER BY total DESC
LIMIT 10
# Next Python cell can access the result
top_customers = _sqldf  # this is the DataFrame from the last %sql cell
top_customers.write.format("delta").saveAsTable("gold.top_customers")
Exercises
  1. Build a parameterized notebook. Create a notebook that accepts two widgets: start_date (text) and env (dropdown: dev/staging/prod). Use the values to build a table path like f"{env}.orders_{start_date}" and print it. Manually change widget values and re-run to verify they update.
  2. dbutils.secrets safety check. In your workspace, list all secret scopes with dbutils.secrets.listScopes(). For each scope, list the available keys. Note that you cannot retrieve values this way — confirm that any hardcoded credentials in your notebooks can be replaced with secret references.
  3. %run vs import. Create two notebooks: one with a helper function defined, and one that calls it using %run. Then refactor the helper into a .py file in a Databricks Repo and import it normally with Python's import. Compare the approaches: which is easier to test? Which has a cleaner namespace?
  4. display() chart. Load any DataFrame with at least one numeric column and one categorical column. Call display(df) and click the "+" chart icon to create a bar chart from the cell output. This is how Databricks ad-hoc dashboards work without any additional tools.
