Architecture & Clusters

What is actually happening when you run a cell. Understanding the machinery underneath Databricks.

The Mental Model: Driver and Executors

Basic PySpark hides a distributed system behind a comfortable Python API. When you write df.groupBy("country").count(), you are describing a computation — not running it. Databricks translates that description into tasks, distributes those tasks across a cluster of machines, and collects the results.

Every Databricks cluster has two types of nodes:

Node TypeRoleWhat lives there
DriverOne per clusterYour notebook process, SparkSession (the entry point your code uses), SparkContext (the lower-level engine it wraps), task scheduler, result collection
ExecutorN per clusterJVM processes that execute tasks, hold partition data in memory, write shuffle files
If You Know Basic PySpark

When you call SparkSession.builder.getOrCreate() locally, the "cluster" is just your laptop — the driver and executor run in the same JVM. On Databricks, the driver is one EC2 instance and executors are other EC2 instances. This is why df.toPandas() is dangerous at scale: it pulls all executor data across the network back to the driver's memory.

The key consequence: your Python code runs on the driver. Spark transformations execute on executors. This means Python variables, loops, and logic that aren't inside a Spark operation run on a single machine. Only what's expressed as a Spark operation distributes.

import pyspark.sql.functions as F  # standard convention: import the functions module as F
# F.col(), F.lit(), F.sum(), etc. — all Spark column expressions live here
# You'll also see "from pyspark.sql.functions import *" in scripts that use many functions

# This loop runs on the DRIVER — not distributed
for country in ["US", "UK", "DE"]:
    df_country = df.filter(F.col("country") == country)
    # Each filter() call creates a Spark plan, but the loop itself is sequential Python

# This is distributed — one Spark job, all workers
df.groupBy("country").count()  # all partitions processed in parallel

Databricks Runtime vs Open-Source Spark

Which DBR Version Should You Use?

Always use an LTS (Long-Term Support) runtime for any work you care about. LTS releases receive security patches and bug fixes for at least 2 years. As of 2025, DBR 15.4 LTS is the current recommended version. When creating a cluster, filter runtimes by "LTS" to see stable options. Non-LTS runtimes are preview/rolling releases — they add new features faster but break more often. The Databricks UI shows a star (★) or "LTS" badge next to long-term-support runtimes.

Databricks doesn't just run vanilla Apache Spark. Databricks Runtime (DBR) is a hardened, optimized distribution that ships with proprietary optimizations on top of the open-source engine:

FeatureOpen-Source SparkDatabricks Runtime
Photon engineNoYes (DBR 9.1+) — C++ vectorized execution
Delta LakeSeparate OSS projectNative, deeply integrated
Auto OptimizeNoYes — automatic small file compaction
Liquid ClusteringNoYes (DBR 13.3+)
Enhanced AQEBasicExtended with skew detection
Pre-installed librariesMinimalpandas, numpy, scikit-learn, MLflow, etc.
What Is a DBU?

A Databricks Unit (DBU) is Databricks' billing unit — a measure of processing capacity consumed per hour. You pay a $/DBU rate that depends on your cluster type (all-purpose vs job cluster), runtime (standard vs Photon), and Databricks tier. The same EC2 instance running as a job cluster costs fewer DBUs/hour than an all-purpose cluster. DBU rates are listed in the Databricks pricing page for your cloud region.

Photon: The Silent Speedup

Photon is a vectorized query engine written in C++ that replaces Spark's JVM-based execution for SQL and DataFrame operations. To use Photon, select a runtime ending in "Photon" when creating a cluster (e.g., "15.4.x-photon-scala2.12" — the current 2025 LTS). It then automatically activates for supported operations — you don't change any code. Most SQL-heavy workloads see 2-8x speedup. The trade-off: Photon costs more DBUs per hour than non-Photon runtimes. Always benchmark before assuming Photon is cheaper end-to-end.

Cluster Types: Which One to Use

Databricks has three distinct compute types, and using the wrong one is expensive:

All-Purpose (Interactive) Clusters

These are the clusters you create from the Compute tab and attach notebooks to. They persist until terminated and are shared across users and notebooks on the same cluster.

  • Use for: development, exploration, iterative analysis
  • Cost gotcha: They keep running (and billing) when idle unless you set auto-termination
  • Best practice: Always set auto termination = 30 minutes on dev clusters

Job Clusters

Ephemeral clusters created by a Job run and terminated when the job finishes. You configure them in the job definition, not the Compute UI.

  • Use for: production pipelines, scheduled jobs, anything that runs unattended
  • Cost advantage: ~30-50% cheaper DBU rate than all-purpose clusters
  • Cold start: 4-8 minutes to provision, so not suitable for low-latency needs

SQL Warehouses (formerly SQL Endpoints)

Purpose-built compute for Databricks SQL. Managed by Databricks, auto-scaling, auto-suspending. Used by Databricks SQL dashboards, SQL Editor, and BI tools via JDBC/ODBC.

  • Use for: BI queries, dashboards, ad-hoc SQL exploration
  • Not for: ETL pipelines, complex Python logic, ML workloads
Common Mistake: All-Purpose Clusters in Production

Running production jobs on all-purpose clusters is expensive and fragile. You pay a higher DBU rate, cluster configuration changes affect all running jobs, and the cluster's uptime is not guaranteed. Always use job clusters for production. The extra cold-start time is almost always worth the cost and reliability improvement.

Cluster Configuration That Matters

Most engineers accept default cluster configs and wonder why things are slow or expensive. Here are the configs that actually matter:

Instance Types

EC2 instances come in families. For Databricks workloads, the typical guidance is:

WorkloadInstance FamilyWhy
General ETL / SQLr5 / r6i (memory-optimized)Spark executors are memory-hungry; shuffles spill to disk when memory is tight
Compute-heavy transformationsc5 / c6i (compute-optimized)CPU-bound operations like sorting, hashing
ML trainingp3 / g4dn (GPU)Deep learning requires GPU cores
Large in-memory datasetsx1e / r6idVery large memory per instance

Spot vs On-Demand Workers

Spot instances are spare EC2 capacity sold at up to 90% discount. Databricks supports spot workers (not drivers) natively:

{
  "driver_node_type_id": "r5.xlarge",
  "node_type_id": "r5.4xlarge",
  "aws_attributes": {
    "availability": "SPOT_WITH_FALLBACK",
    "spot_bid_price_percent": 100
  }
}

SPOT_WITH_FALLBACK is the right setting for most jobs: try spot first, fall back to on-demand if spot isn't available. Pure spot (SPOT) is risky — your job will fail if instances are reclaimed.

Autoscaling

Databricks autoscaling adds and removes workers based on pending tasks. It sounds great but has a subtle problem: Spark doesn't know how many workers it will have when planning a job. This leads to under-partitioned shuffles when workers scale up mid-job.

Autoscaling Trade-off

Enhanced autoscaling (Databricks Runtime Jobs clusters only) solves the re-partitioning problem with cluster-level optimizations: Spark can run efficiently even as worker count changes mid-job, and the cluster scales down more aggressively by removing idle workers faster. Basic autoscaling (default on open-source Spark) uses a simple task-queue heuristic and can actually increase job duration — new workers arrive after large shuffles are already planned at lower parallelism. For jobs with predictable data volumes, a fixed worker count is often faster and cheaper than either mode. Enable Enhanced by checking "Enable autoscaling" on a Jobs cluster (it's distinct from the autoscaling on All-Purpose clusters).

Storage: DBFS vs External Storage

DBFS (Databricks File System) is a virtual file system mounted on top of your cloud storage. It makes S3 paths look like local filesystem paths:

# DBFS path (abstraction over S3)
df = spark.read.parquet("dbfs:/mnt/my-bucket/data/")

# Direct S3 path (requires configured access)
df = spark.read.parquet("s3://my-bucket/data/")

# Unity Catalog Volume path (preferred in modern Databricks)
df = spark.read.parquet("/Volumes/catalog/schema/volume/data/")
DBFS Mounts Are Deprecated

The old pattern of mounting S3 buckets to DBFS with dbutils.fs.mount() is deprecated in Unity Catalog workspaces. The modern approach is External Locations in Unity Catalog (Chapter 11) and Volumes for unstructured data. Avoid creating new mounts — migrate existing ones when you adopt Unity Catalog.

What Lives in DBFS Root

The DBFS root (dbfs:/) maps to a Databricks-managed S3 bucket in your AWS account. You should treat it as scratch space only. User data, Delta tables, and anything you care about should live in your own S3 buckets, accessed via Unity Catalog External Locations.

Path TypeExampleUse For
DBFS rootdbfs:/tmp/, dbfs:/user/Temporary files, local scratch — not production data
DBFS mount (deprecated)dbfs:/mnt/my-bucket/Legacy access to S3; avoid in new workspaces
Direct S3s3://my-bucket/path/Direct access with instance profile auth

The Spark UI: Your First Debugging Tool

Every Databricks cluster has a Spark UI accessible from the cluster detail page. It's the most important performance debugging tool you have:

  • Jobs tab — every action (collect, write, count) creates a job; shows duration and stage breakdown
  • Stages tab — each stage is a set of tasks separated by a shuffle; shows spill and GC metrics
  • SQL/DataFrame tab — visualized query plan with stage timings
  • Storage tab — cached DataFrames/tables, memory usage
  • Executors tab — per-executor memory, task time, GC overhead, shuffle read/write
Actions vs Transformations

Transformations (filter, select, join) are lazy — they build a logical plan without executing anything. Actions (count, collect, show, write) trigger actual execution. This is why you see one Spark job per action in the Spark UI, not one per transformation. Understanding this distinction is the single most important concept for reasoning about Spark performance.

Quick Check

You have a job cluster with 10 spot workers running a 4-hour ETL job. Two hours in, AWS reclaims 3 spot instances. What happens?

  • The job fails immediately
  • The job pauses until replacement instances are available
  • Databricks provisions replacement instances and the job continues, possibly slower
  • The driver node takes over and runs the job single-threaded
Databricks provisions replacements and continues. With SPOT_WITH_FALLBACK, Databricks automatically requests replacement instances. Tasks that were running on reclaimed instances are rescheduled. There may be a performance dip while new instances initialize, but the job does not fail. If you use pure SPOT and AWS can't provide replacements quickly, the job may stall or fail.
Getting Sample Data for the Exercises

Every Databricks workspace ships with built-in sample datasets — no setup needed. In a notebook cell, run:

diamonds = spark.read.csv("/databricks-datasets/diamonds/diamonds.csv",
                          header=True, inferSchema=True)
diamonds.printSchema()
diamonds.show(5)

Other useful built-in datasets: /databricks-datasets/nyctaxi/ (NYC taxi trips, ~1.5GB), /databricks-datasets/retail-org/ (retail orders), /databricks-datasets/wikipedia-datasets/. Browse all of them: display(dbutils.fs.ls("/databricks-datasets/"))

Exercises
  1. Identify your compute types. In the left sidebar, click Compute. In the current Databricks UI (2024+), this page has two sections: All-purpose clusters (interactive notebooks) and SQL Warehouses (SQL Editor, Dashboards). If you only see SQL Warehouses and no all-purpose clusters, that's normal for a fresh or SQL-focused workspace — you'll need to create an All-Purpose cluster to run notebook code. For any cluster listed: check its state (Running/Terminated), its DBR version, and whether auto-termination is configured (shown as "Terminates after N minutes of inactivity"). SQL Warehouses auto-suspend independently — check the Suspend setting under the warehouse's Edit page.
  2. Read the Spark UI. First, attach a notebook to an All-Purpose cluster (not a SQL Warehouse). Load the diamonds dataset and run: 
    diamonds = spark.read.format("csv").option("header", True).option("inferSchema", True) \
    .load("/databricks-datasets/diamonds/diamonds.csv")
diamonds.groupBy("cut").count().show()
    Finding the Spark UI: After the cell runs, look for the blue Spark Jobs link directly below the cell output (it says "Job N, N tasks"). Click it — this opens the Spark UI for that specific job. Alternatively: left sidebar → Compute → click your cluster name → Spark UI tab at the top. In the Spark UI, go to Stages and identify: how many stages ran? Look for the "Shuffle Read/Write" column — a non-zero value means a shuffle happened. How many tasks ran per stage?
  3. Driver vs executor exercise. Run both approaches below and compare timing with %%timeit. The loop approach calls Spark 1,000 times (one union per row, all on the driver). The batch approach makes one Spark call. Why is the loop slower even on the same hardware? 
    # Loop approach: 1,000 driver-side Spark calls
from pyspark.sql import Row
df = spark.createDataFrame([], "id INT")
for i in range(1000):
    df = df.union(spark.createDataFrame([Row(id=i)]))
df.count()

# Batch approach: one Spark call
df2 = spark.createDataFrame([(i,) for i in range(1000)], ["id"])
df2.count()
  4. Estimate cluster cost. Go to databricks.com/pricing (this is the public Databricks marketing website — not your workspace console). Select your cloud provider (AWS/Azure/GCP). Find the DBU rate for All-Purpose compute, Standard tier. Then estimate: if your dev cluster runs 8 hours/day for 20 working days, what does it cost in DBUs? Now configure auto-termination at 30 minutes idle (Compute → your cluster → Edit → Auto Termination) and re-estimate assuming 50% idle time. The difference is usually significant. Note: DBU prices are listed at databricks.com/pricing; your actual cloud bill also includes VM costs (EC2/VMs) which are separate.
