Chapter 12

AWS Integration

Instance profiles (not access keys), Databricks secrets, S3 access patterns, and cross-account IAM. How Databricks and AWS talk to each other.

Rule #1: Never Use Access Keys in Notebooks

This deserves its own section because it's the most common security mistake on Databricks. Access keys (aws_access_key_id / aws_secret_access_key) hardcoded in notebooks or stored in variables are a critical security risk:

# NEVER DO THIS
spark.conf.set("fs.s3a.access.key", "AKIAIOSFODNN7EXAMPLE")       # visible in logs
spark.conf.set("fs.s3a.secret.key", "wJalrXUtnFEMI/K7MDENG/bPxR")  # visible in Git history

# ALSO AVOID (still exposed in notebook state)
import boto3
s3 = boto3.client(
    's3',
    aws_access_key_id="AKIAIOSFODNN7EXAMPLE",  # don't do this
    aws_secret_access_key="wJalrXUtnFEMI/K7MDENG/bPxR"
)
Access Keys in Notebooks = Security Incident

Databricks notebooks are version-controlled and shared across teams. Access keys in notebooks are almost always leaked: into Git history, into run logs, into screenshots, into shared notebook links. They also expire unpredictably. Use instance profiles (cluster-attached IAM roles) or Databricks secrets instead. Both are zero-code-change approaches once configured by an admin.

Instance Profiles: The Right Way

An instance profile is an IAM role attached to EC2 instances (your cluster workers). All code on those instances inherits the role's permissions automatically — no credentials in code:

# With instance profile attached to the cluster: just use the path
df = spark.read.parquet("s3://my-secure-bucket/data/")

# boto3 also inherits the role automatically
import boto3
s3 = boto3.client('s3')  # no credentials needed — IAM role is implicit
s3.get_object(Bucket="my-secure-bucket", Key="config/settings.json")

Instance profiles are configured by Databricks admins at the workspace level, then attached to clusters at cluster creation. No code changes needed in notebooks.

Databricks Secrets: For Non-IAM Credentials

For credentials that aren't IAM-controlled (API keys, database passwords, third-party service tokens), use Databricks secrets:

# Set up a secret scope and secrets using the Databricks CLI
databricks secrets create-scope --scope my-api-scope
databricks secrets put --scope my-api-scope --key stripe-api-key
databricks secrets put --scope my-api-scope --key postgres-password

# List scopes and keys (CLI)
databricks secrets list-scopes
databricks secrets list --scope my-api-scope
# In notebooks: retrieve and use
stripe_key = dbutils.secrets.get(scope="my-api-scope", key="stripe-api-key")
db_password = dbutils.secrets.get(scope="my-api-scope", key="postgres-password")

# Use in connection string
jdbc_url = f"jdbc:postgresql://my-db.cluster.us-east-1.rds.amazonaws.com:5432/orders"
df = spark.read \
    .format("jdbc") \
    .option("url", jdbc_url) \
    .option("dbtable", "orders") \
    .option("user", "app_user") \
    .option("password", db_password) \  # secret value — redacted in cell output
    .load()
AWS Secrets Manager-backed Scopes

Databricks secrets can be backed by AWS Secrets Manager. With an AWS Secrets Manager-backed scope, secrets created in AWS Secrets Manager are automatically available in Databricks without duplicating them. This is the recommended approach for organizations already using AWS Secrets Manager as their secrets management platform.

Cross-Account S3 Access

Databricks clusters live in your AWS account, but data may be in a different AWS account. Cross-account access requires both a bucket policy (in the data account) and an IAM role trust (in the Databricks account):

// S3 bucket policy in the data account
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "AWS": "arn:aws:iam::DATABRICKS-ACCOUNT-ID:role/databricks-instance-profile"
    },
    "Action": ["s3:GetObject", "s3:PutObject", "s3:ListBucket"],
    "Resource": [
      "arn:aws:s3:::cross-account-data-bucket",
      "arn:aws:s3:::cross-account-data-bucket/*"
    ]
  }]
}
# Once the IAM/bucket policy is set, access works transparently
df = spark.read.parquet("s3://cross-account-data-bucket/data/2024/")
# No code change — instance profile handles cross-account auth

VPC and Network Considerations

Databricks clusters run in your VPC. For private S3 access (traffic never leaves AWS network):

  • VPC Endpoint for S3: Route S3 traffic through the VPC endpoint instead of public internet. Required for security-sensitive workloads.
  • PrivateLink: For the Databricks control plane connection. Prevents traffic from leaving AWS.
  • Security groups: Clusters need egress to S3 (port 443) and to the Databricks control plane.
S3 Transfer Costs

Data transfer out of AWS to the internet costs money. Data transfer between S3 and EC2 in the same region is free. Always ensure your Databricks workspace and S3 buckets are in the same AWS region. Cross-region S3 access works but incurs data transfer costs of ~$0.02/GB and higher latency. When designing multi-region architectures, replicate data to the local region first.

Exercises
  1. Secret scope audit. List all secret scopes in your workspace with dbutils.secrets.listScopes(). For each scope, list the keys. Identify any notebooks that hardcode credentials (grep your repo for access_key, password =, token = in plain text) and plan a migration to secrets.
  2. Instance profile verification. If your cluster has an instance profile attached, verify it works: read a file from S3 using a direct path (s3://your-bucket/test-file) and confirm no credential configuration is needed in code. If you don't have an instance profile, document what IAM role would need to exist and what policies it would need.
  3. Secret vs environment variable. Retrieve a secret value with dbutils.secrets.get() and try to print() it in a cell. Confirm that Databricks redacts it in the cell output (shows [REDACTED]). Then set the same value as an environment variable and print it — observe that it's NOT redacted. Document why you'd always prefer secrets over environment variables.
