Chapter 9

Catalyst Optimizer & AQE

How Spark transforms your query into an execution plan. Understanding Catalyst lets you write queries that optimize well — and recognize ones that don't.

The Catalyst Pipeline

Every SQL query or DataFrame transformation goes through four stages before execution:

  1. Parsing — SQL text → unresolved logical plan (AST — Abstract Syntax Tree, a tree representation of your query's structure where table and column names are not yet validated against the catalog)
  2. Analysis — resolve table names, column names, types against the catalog → analyzed logical plan
  3. Logical Optimization — apply rule-based transformations (predicate pushdown, column pruning, constant folding) → optimized logical plan
  4. Physical Planning — choose physical operators (which join algorithm?), cost-based join strategy selection → physical plan
SQL text or DataFrame API
        ↓  Parsing
Unresolved Logical Plan (AST — table/column names not yet resolved)
        ↓  Analysis (catalog lookup)
Analyzed Logical Plan (types and names resolved)
        ↓  Rule-Based Optimization (predicate pushdown, column pruning, etc.)
Optimized Logical Plan
        ↓  Physical Planning (join strategies, exchange placement)
Physical Plan → Execution (Spark tasks on executors)

You can see all four in EXPLAIN EXTENDED:

spark.sql("""
    SELECT customer_id, COUNT(*), SUM(amount)
    FROM orders
    WHERE status = 'completed' AND amount > 0
    GROUP BY customer_id
    HAVING SUM(amount) > 500
""").explain("extended")

# Output sections:
# == Parsed Logical Plan ==        (what you wrote, unresolved)
# == Analyzed Logical Plan ==      (columns/types resolved)
# == Optimized Logical Plan ==     (after rule-based optimization)
# == Physical Plan ==              (actual execution operators)

Rule-Based Optimizations

Catalyst applies dozens of transformation rules automatically. The most impactful ones:

Predicate Pushdown

Filters move as close to the data source as possible. A WHERE status = 'completed' that appears after a join gets pushed to the scan node:

-- What you write:
SELECT o.order_id, c.email
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
WHERE o.status = 'completed';

-- What Catalyst optimizes to (logically):
SELECT o.order_id, c.email
FROM (SELECT * FROM orders WHERE status = 'completed') o
JOIN customers c ON o.customer_id = c.customer_id;
-- The filter runs during the scan, not after the join
-- Fewer rows enter the join = much less shuffle data

Column Pruning (Projection Pushdown)

Catalyst reads only the columns needed, not SELECT *:

-- What you write:
SELECT customer_id, SUM(amount)
FROM orders
GROUP BY customer_id;

-- Catalyst reads only [customer_id, amount] from Parquet
-- Not the 20 other columns in the table
-- This is why columnar formats (Parquet/Delta) are fast for analytics

Constant Folding and Simplification

-- Catalyst evaluates these at plan time, not per row:
WHERE 1 + 1 = 2           -- becomes TRUE (whole table)
WHERE amount * 1.0 > 100  -- simplifies to amount > 100.0
WHERE amount BETWEEN 0 AND 1000  -- becomes amount >= 0 AND amount <= 1000

Adaptive Query Execution (AQE)

AQE (Spark 3.0+, enabled by default on Databricks) re-optimizes query plans at runtime based on actual data statistics collected during execution. It overcomes the fundamental limitation of static planning: the optimizer doesn't know actual data sizes until the job starts running.

AQE kicks in after each shuffle stage completes, using the actual partition sizes and row counts to make better decisions for subsequent stages.

Dynamic Partition Coalescing

After a shuffle, AQE coalesces many small shuffle partitions into fewer, larger ones. This eliminates the "many tiny tasks" problem for queries that produce less data than expected:

# Without AQE: 200 shuffle partitions (default), most nearly empty after filtering
# 200 tasks each reading 1KB = massively inefficient

# With AQE: detects small partitions and coalesces
# 200 partitions → 8 coalesced partitions → 8 tasks each reading 25MB
# Net: 25x fewer tasks, much less scheduling overhead

# Control AQE coalescing
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")  # default
spark.conf.set("spark.sql.adaptive.advisoryPartitionSizeInBytes", "67108864")  # 64MB target

Dynamic Join Strategy Selection

AQE can switch from sort-merge to broadcast join at runtime if a join side turns out to be smaller than expected after filtering:

# Query: join orders to customers, filtered
# Static plan chose sort-merge (customers is 500MB before filter)
# But WHERE region = 'US' reduces customers to 5MB

# AQE detects the actual size after the filter executes
# and switches to broadcast join at runtime
# No hint needed — AQE does this automatically

# Monitor in the Spark UI: look for "AQE" annotations in the SQL tab
# "dynamically determined broadcast" in the plan

spark.conf.set("spark.sql.adaptive.enabled", "true")  # default on Databricks

AQE Skew Join Handling

# AQE automatically detects skewed partitions (one partition >> median size)
# and splits them into multiple tasks

spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")  # default
spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionFactor", "5")  # partition > 5x median = skewed
spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionThresholdInBytes", "268435456")  # 256MB min to trigger
AQE Limitations

AQE cannot help with: (1) skew in the first stage (before any shuffle), (2) broadcast OOM caused by a table being larger than executor memory, (3) queries with no shuffles (nothing to re-plan). For skew before the first shuffle, use salting (Chapter 8). AQE is also less effective when statistics are completely absent — it re-plans based on actual sizes but can't predict what the first stage will produce.

Key Configuration Knobs

Three Ways to Set Spark Config — When to Use Each

You'll see configs set three different ways throughout this guide:

  • spark.conf.set("key", "value") — Python/PySpark, session-scoped. Only affects the current SparkSession (your notebook or job run). Reset on cluster restart. Use this in notebooks for per-query tuning.
  • SET key = value; — SQL, session-scoped. Equivalent to spark.conf.set() but in a %sql cell. Same scope, same lifetime.
  • Cluster JSON config (Spark Config field in the cluster UI) — Cluster-scoped. Applies to every session on that cluster, persists across restarts. Use this for permanent cluster-wide defaults (like enabling Photon or setting default shuffle partitions for a team's cluster). Set via Cluster UI → Advanced Options → Spark Config.

Rule of thumb: query-level tuning → spark.conf.set(); team/cluster-wide defaults → cluster JSON config.

ConfigDefaultWhen to Change
spark.sql.shuffle.partitions200Decrease for small datasets (<10GB); AQE overrides this anyway
spark.sql.autoBroadcastJoinThreshold10MBIncrease to 50-100MB if dimension tables are larger but small enough to broadcast safely

spark.sql.adaptive.coalescePartitions.minPartitionNum1Set to cluster executor count to prevent under-parallelism after coalescing

Exercises
  1. Trace the optimization pipeline. Write a moderately complex query (2 joins, a GROUP BY, a HAVING clause). Run EXPLAIN EXTENDED and compare the Optimized Logical Plan to the Parsed Logical Plan. Identify at least two optimizations Catalyst applied (predicate pushdown, column pruning, join reordering, etc.).
  2. Observe AQE in action. Run a query that filters one join side to a very small result before the join. Check the SQL tab in the Spark UI for any AQE annotations ("dynamically converted to broadcast"). Compare the plan with and without spark.sql.adaptive.enabled = false.
  3. Tune shuffle partitions. Run an aggregation on a small table (under 1GB). Check how many shuffle partitions AQE coalesced to. Then disable AQE and set shuffle.partitions = 8 manually to match. Which approach produced better task utilization?
