Chapter 8

Join Strategies

Broadcast, sort-merge, shuffle hash, data skew, and salting. Joins are the most common source of performance problems in Spark workloads.

Join Strategies Overview

Spark has several physical join implementations. The optimizer chooses based on table sizes, statistics, and join type. Understanding when each is chosen and why tells you how to optimize a slow join:

StrategyWhen UsedShuffle Required?Best For
Broadcast Hash JoinOne side < broadcast thresholdNo — small side broadcast to all workersLarge table + small dimension table
Sort-Merge JoinBoth sides largeYes — both sides shuffled by key, then sortedLarge-large joins, equi-joins
Shuffle Hash JoinOne side fits in executor memoryYes — one side shuffled and hashedSkewed data, unsortable keys
Broadcast Nested LoopNon-equi join (>, !=, CROSS)No for small sideInequality conditions; avoid on large tables
Cartesian ProductCROSS JOIN or no join conditionNoVery small tables only

Broadcast Hash Join

The fastest join type. The small table is serialized and sent to every executor, avoiding any shuffle of the large table:

-- Auto-broadcast: table smaller than spark.sql.autoBroadcastJoinThreshold (default 10MB)
SELECT o.*, c.email, c.country
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id;
-- If customers < 10MB, Spark auto-broadcasts it

-- Manual hint: force broadcast even above threshold
SELECT /*+ BROADCAST(c) */ o.*, c.email
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id;

-- Increase auto-broadcast threshold (session level)
SET spark.sql.autoBroadcastJoinThreshold = 52428800;  -- 50MB
# PySpark broadcast hint
from pyspark.sql.functions import broadcast

result = orders.join(
    broadcast(customers),
    on="customer_id",
    how="inner"
)
Broadcast OOM Risk

Broadcasting a table uses driver memory (to serialize) and executor memory on each node (to hold the hash table). If you broadcast a 200MB table to 20 executors with 4GB memory each, that's 4GB just for this join. Databricks will OOM if the broadcast size exceeds executor memory. Check table size with DESCRIBE DETAIL before forcing a broadcast hint above the default threshold.

Sort-Merge Join

The default for large-large joins. Both tables are shuffled by join key, sorted, and then merged. It requires a shuffle but is predictable and memory-efficient:

# Observe sort-merge join in action
df = spark.sql("""
    EXPLAIN FORMATTED
    SELECT o.order_id, o.amount, c.email
    FROM orders o  -- 500GB
    JOIN customers c  -- 50GB
    ON o.customer_id = c.customer_id
""")
# Physical plan will show: SortMergeJoin [customer_id], [customer_id], Inner
# With Exchange (shuffle) on both sides

Sort-Merge join is often acceptable. Optimize it by:

  • Reducing shuffle partitions: SET spark.sql.shuffle.partitions = 200 (default is 200 but AQE adjusts this)
  • Ensuring join columns are the same type (type mismatch forces a cast which prevents pushdown)
  • Pre-sorting the data if you join on the same key repeatedly

Data Skew: The Silent Killer

Skew occurs when one join key value contains a disproportionate number of rows. One executor gets stuck processing the "hot key" while all others finish and sit idle. This is the most common cause of a job that runs to 99% and then hangs.

Symptoms: one task takes 10x longer than all others in the Spark UI Stages tab.

# Diagnose skew: check key distribution
spark.sql("""
    SELECT customer_id, COUNT(*) AS cnt
    FROM orders
    GROUP BY customer_id
    ORDER BY cnt DESC
    LIMIT 20
""").display()
# If top key has 10M rows and median has 100 rows, you have skew

Fixing Skew with Salting

Salting artificially increases cardinality of the hot key by adding a random suffix. This distributes the skewed key across N sub-keys, each processed by a different executor:

import pyspark.sql.functions as F

N_BUCKETS = 10  # number of salt values — each hot key becomes 10 sub-keys

# Step 1: Salt the LARGE (skewed) side with a random bucket 0..9
# (F.rand() * N_BUCKETS).cast("int") gives a random int in [0, N_BUCKETS-1]
orders_salted = orders.withColumn(
    "salted_customer_id",
    F.concat(F.col("customer_id"), F.lit("_"), (F.rand() * N_BUCKETS).cast("int").cast("string"))
)

# Step 2: Explode the SMALL side to match all possible salt values
# We crossJoin customers with [0, 1, 2, ..., N_BUCKETS-1] so each customer gets N_BUCKETS rows
# (one per possible bucket). This is necessary because an order with salt "_3" must join
# to the customer row that also has salt "_3" — the small side needs to be in every bucket.
# crossJoin on a tiny table (customers) is cheap even if it increases its size N times.
customers_exploded = customers \
    .crossJoin(spark.range(N_BUCKETS).select(F.col("id").cast("string").alias("salt"))) \
    .withColumn(
        "salted_customer_id",
        F.concat(F.col("customer_id"), F.lit("_"), F.col("salt"))
    )

# Step 3: Join on the salted key — hot key "C001" is now spread across 10 sub-keys
result = orders_salted.join(
    customers_exploded.drop("salt"),
    on="salted_customer_id",
    how="inner"
).drop("salted_customer_id")
AQE Skew Handling

Databricks' Adaptive Query Execution (AQE) includes automatic skew join handling. When AQE detects that one join partition is much larger than the median, it splits that partition into smaller sub-partitions automatically. Enable with spark.sql.adaptive.skewJoin.enabled = true (default: true on Databricks). AQE handles most common skew cases. Use manual salting only when AQE isn't sufficient.

Join Ordering

When joining multiple tables, the order matters for performance. Join the most selective (most filtering) tables first, and broadcast small dimension tables early to reduce the size of intermediate results:

-- Inefficient: cross product of large tables before filtering
SELECT *
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id   -- 500GB x 2TB = pain
JOIN products p ON oi.product_id = p.product_id
WHERE o.status = 'completed'
  AND p.category = 'electronics';

-- Better: filter first, broadcast small dimension
SELECT /*+ BROADCAST(p) */  *
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
JOIN products p ON oi.product_id = p.product_id
WHERE o.status = 'completed'    -- applies early via predicate pushdown
  AND p.category = 'electronics';

-- Even better: pre-filter in a CTE
WITH filtered_orders AS (
    SELECT * FROM orders WHERE status = 'completed'
),
electronics_products AS (
    SELECT * FROM products WHERE category = 'electronics'
)
SELECT /*+ BROADCAST(ep) */  *
FROM filtered_orders fo
JOIN order_items oi ON fo.order_id = oi.order_id
JOIN electronics_products ep ON oi.product_id = ep.product_id;
CBO and Join Reordering

With accurate table statistics (ANALYZE TABLE ... COMPUTE STATISTICS FOR ALL COLUMNS), Spark's cost-based optimizer can automatically reorder joins to minimize intermediate result sizes. Enable with SET spark.sql.cbo.enabled = true (default: true on Databricks). CBO works best when statistics are fresh and representative — stale statistics lead to poor ordering choices.

Quick Check

A job is joining a 500GB orders table to a 500MB customers table. The job hangs at 99% for 30 minutes. The Spark UI shows one task taking 25x longer than all others. What's the most likely cause and fix?

  • The cluster is running out of disk space; add more workers
  • Data skew on the join key; fix with AQE skew join or manual salting
  • The customers table is too large to broadcast; increase the broadcast threshold to 1GB
  • The join type should be LEFT OUTER instead of INNER; rewrite the query
Data skew. One task running 25x longer than others is the classic skew signature: one join key value has far more rows than others. First check if AQE skew handling is enabled (spark.sql.adaptive.skewJoin.enabled). If it is but skew is still happening, the hot-key partition may exceed AQE's skew detection thresholds. Investigate key distribution and apply salting if necessary. Broadcasting customers (500MB) is risky for memory reasons and doesn't address skew.
