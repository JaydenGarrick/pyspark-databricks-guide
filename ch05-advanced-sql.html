Advanced SQL Patterns

CTEs, PIVOT, array and struct operations, JSON parsing, and EXECUTE IMMEDIATE. The SQL toolkit for complex transformations.

CTEs — Common Table Expressions

CTEs give names to subqueries so they can be referenced multiple times. They make complex queries readable by breaking them into named, logical steps:

WITH
-- Step 1: Calculate customer lifetime value
customer_ltv AS (
    SELECT
        customer_id,
        SUM(amount) AS total_spent,
        COUNT(*) AS order_count,
        MIN(created_at) AS first_order,
        MAX(created_at) AS last_order
    FROM orders
    WHERE status = 'completed'
    GROUP BY customer_id
),
-- Step 2: Assign LTV segments
ltv_segments AS (
    SELECT
        customer_id,
        total_spent,
        order_count,
        CASE
            WHEN total_spent >= 1000 THEN 'platinum'
            WHEN total_spent >= 500  THEN 'gold'
            WHEN total_spent >= 100  THEN 'silver'
            ELSE 'bronze'
        END AS ltv_segment
    FROM customer_ltv
),
-- Step 3: Join to customer profile
enriched AS (
    SELECT
        c.customer_id,
        c.email,
        c.country,
        s.ltv_segment,
        s.total_spent,
        s.order_count
    FROM customers c
    JOIN ltv_segments s USING (customer_id)
)
-- Final SELECT against the last CTE
SELECT ltv_segment, country, COUNT(*) AS customers, AVG(total_spent) AS avg_ltv
FROM enriched
GROUP BY ltv_segment, country
ORDER BY ltv_segment, customers DESC;
If You Know Basic PySpark

The PySpark equivalent is chaining .alias() and assigning intermediate DataFrames to variables. CTEs are often cleaner for pure analytics. However, for reusable logic and complex conditional transforms, the DataFrame API gives you Python functions, loops, and types that SQL cannot. Use SQL for query-shaped work, Python for pipeline-shaped work.

CTE Materialization

In Spark SQL, CTEs are not automatically materialized. If you reference a CTE three times, Spark may re-execute it three times. For expensive CTEs you need multiple times, either: (1) use spark.sql("CREATE OR REPLACE TEMP VIEW my_cte AS ...") to persist it, or (2) cache the intermediate result as a Delta temp table. Note: some other SQL engines do materialize CTEs automatically; Spark does not.

PIVOT

PIVOT rotates row values into columns — turning a "tall" table into a "wide" one:

-- Input: one row per (month, region, revenue)
-- Output: one row per month, one column per region

SELECT *
FROM monthly_revenue
PIVOT (
    SUM(revenue)            -- aggregate function
    FOR region              -- column whose values become new column names
    IN ('US', 'EU', 'APAC') -- the values to pivot into columns
);
-- Result columns: month, US, EU, APAC
-- More complex: pivot with multiple aggregates
SELECT *
FROM orders
PIVOT (
    SUM(amount)  AS sum_amount,
    COUNT(*)     AS order_count
    FOR status
    IN ('pending' AS pending, 'completed' AS completed, 'cancelled' AS cancelled)
);
Dynamic PIVOT in PySpark

Spark SQL's PIVOT requires you to know the column values at write time. For dynamic pivots (unknown values), use the PySpark DataFrame API: collect the distinct values first, then build the pivot expression programmatically. df.groupBy("month").pivot("region").sum("revenue") — the DataFrame API will scan for distinct values automatically, but this adds an extra scan pass.

Arrays and Structs

Delta tables frequently contain nested data: arrays (lists), structs (objects), and maps. Spark SQL has rich support for working with these types without flattening:

Array Operations

-- Sample: orders table with items array
-- items: ARRAY<STRUCT<product_id STRING, qty INT, price DECIMAL>>

-- Size of array
SELECT order_id, SIZE(items) AS item_count FROM orders;

-- Check if array contains a value
SELECT * FROM orders
WHERE ARRAY_CONTAINS(items.product_id, 'P001');

-- Flatten array elements to rows (EXPLODE)
SELECT order_id, item.product_id, item.qty, item.price
FROM orders
LATERAL VIEW EXPLODE(items) AS item;

-- EXPLODE_OUTER preserves rows with empty arrays (returns NULL instead of dropping)
SELECT order_id, item.product_id
FROM orders
LATERAL VIEW OUTER EXPLODE(items) AS item;

-- Transform array elements (equivalent of map())
SELECT
    order_id,
    TRANSFORM(items, item -> item.price * item.qty) AS line_totals
FROM orders;

-- Filter array elements
SELECT
    order_id,
    FILTER(items, item -> item.price > 10) AS expensive_items
FROM orders;

-- Aggregate array
SELECT
    order_id,
    AGGREGATE(items, 0D, (acc, item) -> acc + item.price * item.qty) AS total
    -- 0D is a double literal (0.0 as DOUBLE) — required because price is DECIMAL and
    -- the accumulator type must match the return type
FROM orders;

Struct Operations

-- Access nested struct fields with dot notation
SELECT
    order_id,
    shipping_address.street,
    shipping_address.city,
    shipping_address.zip_code
FROM orders;

-- Reconstruct a struct
SELECT
    order_id,
    STRUCT(
        shipping_address.city AS city,
        shipping_address.country AS country
    ) AS location
FROM orders;

-- Rename/reorder struct fields
SELECT
    order_id,
    NAMED_STRUCT(
        'full_city',  CONCAT(shipping_address.city, ', ', shipping_address.state),
        'zip',        shipping_address.zip_code
    ) AS location_summary
FROM orders;

JSON Parsing

Raw data often arrives as JSON strings in a column. Spark SQL can parse and query JSON without flattening to a separate table:

-- Parse a JSON string column
SELECT
    event_id,
    GET_JSON_OBJECT(payload, '$.user.id') AS user_id,
    GET_JSON_OBJECT(payload, '$.event_type') AS event_type,
    GET_JSON_OBJECT(payload, '$.metadata.source') AS source
FROM raw_events;

-- Parse to a struct with FROM_JSON
SELECT
    event_id,
    FROM_JSON(payload, 'STRUCT<user_id STRING, event_type STRING, amount DECIMAL(10,2)>') AS parsed
FROM raw_events;

-- Then access fields from the parsed struct
SELECT
    event_id,
    parsed.user_id,
    parsed.event_type,
    parsed.amount
FROM (
    SELECT
        event_id,
        FROM_JSON(payload, 'STRUCT<user_id STRING, event_type STRING, amount DECIMAL(10,2)>') AS parsed
    FROM raw_events
)
WHERE parsed.event_type = 'purchase';

-- Convert a struct back to JSON
SELECT order_id, TO_JSON(shipping_address) AS address_json FROM orders;
Schema Inference for JSON

If you don't know the JSON schema upfront, use PySpark's spark.read.option("inferSchema", "true").json(path) to infer it from a sample, then use the inferred schema in FROM_JSON. For large, variable JSON payloads, store in Delta as a string column and parse at query time using FROM_JSON or GET_JSON_OBJECT — this is often better than enforcing a rigid schema upfront.

LATERAL VIEW and INLINE

-- LATERAL VIEW EXPLODE: turn array into rows (row-per-element)
SELECT order_id, tag
FROM orders
LATERAL VIEW EXPLODE(tags) tags_table AS tag;

-- LATERAL VIEW EXPLODE with map: turns map into key-value rows
SELECT order_id, attr_key, attr_value
FROM orders
LATERAL VIEW EXPLODE(attributes) attrs AS attr_key, attr_value;

-- INLINE: explode an array of structs into multiple columns (one column per struct field)
-- Unlike EXPLODE which returns one column containing the struct,
-- INLINE expands each struct field into its own named column.
SELECT order_id, product_id, qty, price
FROM orders
LATERAL VIEW INLINE(items) item_table AS product_id, qty, price;

-- POSEXPLODE: like EXPLODE but also returns element position (0-indexed)
SELECT order_id, pos, tag
FROM orders
LATERAL VIEW POSEXPLODE(tags) tags_table AS pos, tag;

EXECUTE IMMEDIATE (Dynamic SQL)

Version Requirements

EXECUTE IMMEDIATE requires Databricks Runtime 12.2+. The DECLARE variable syntax requires DBR 14.1+ or Databricks SQL. These are Databricks-specific SQL extensions — they are not available in open-source Spark or standard ANSI SQL environments.

EXECUTE IMMEDIATE runs a SQL string at runtime. This enables parameterized SQL and dynamic table/schema names:

-- Basic dynamic SQL
DECLARE table_name STRING DEFAULT 'orders';
EXECUTE IMMEDIATE 'SELECT COUNT(*) FROM ' || table_name;

-- With parameters (prevents SQL injection)
DECLARE customer_id_param STRING DEFAULT 'C001';
EXECUTE IMMEDIATE
    'SELECT * FROM orders WHERE customer_id = ?'
    USING customer_id_param;

-- Build and run table creation dynamically
DECLARE env STRING DEFAULT 'prod';
EXECUTE IMMEDIATE
    'CREATE TABLE IF NOT EXISTS ' || env || '.orders_archive
     USING DELTA AS SELECT * FROM ' || env || '.orders WHERE created_at < current_date() - 365';
# Python equivalent using spark.sql() with string formatting
# NOTE: be careful with string formatting — use parameterized queries when possible
env = "prod"
table_name = dbutils.widgets.get("table_name")

# Safer: use spark.sql() with a prepared statement approach
spark.sql(f"""
    SELECT *
    FROM {env}.{table_name}
    WHERE event_date = '{date_param}'
""")

# Even safer: validate inputs first
allowed_tables = ["orders", "customers", "products"]
if table_name not in allowed_tables:
    raise ValueError(f"Invalid table name: {table_name}")
spark.sql(f"SELECT COUNT(*) FROM {env}.{table_name}")
SQL Injection in Dynamic SQL

Dynamic SQL with string concatenation is vulnerable to injection. Always validate inputs against an allowlist when building table/column names dynamically. For value parameters, use the USING clause in EXECUTE IMMEDIATE or parameterized spark.sql() calls. Never concatenate raw user input directly into a SQL string without validation.

Set Operations

-- UNION ALL: combine all rows from both queries (no dedup)
SELECT customer_id FROM active_customers
UNION ALL
SELECT customer_id FROM churned_customers;

-- UNION (DISTINCT): deduplicate combined result
SELECT customer_id FROM active_customers
UNION
SELECT customer_id FROM churned_customers;

-- INTERSECT: rows in both queries
SELECT customer_id FROM high_value_customers
INTERSECT
SELECT customer_id FROM recent_purchasers;

-- EXCEPT (MINUS): rows in first query but not second
SELECT customer_id FROM all_customers
EXCEPT
SELECT customer_id FROM customers_who_purchased_last_90_days;
UNION ALL vs UNION Performance

UNION ALL is always faster than UNION because UNION adds a deduplication step (essentially a GROUP BY across all columns). Use UNION ALL by default, and only use UNION when deduplication is semantically required. In ETL contexts, UNION ALL is almost always the right choice.

Exercises
  1. Multi-step CTE pipeline. Write a query with at least 4 CTEs that: (1) filters raw orders to completed status, (2) calculates per-product revenue, (3) ranks products by revenue within each category, (4) returns only the top 3 products per category. Use QUALIFY in the final CTE instead of a WHERE on the rank column.
  2. PIVOT a time-series. Given daily revenue data with columns event_date, region, revenue, write a PIVOT query that produces one column per month (Jan, Feb, Mar...) showing total revenue by region. How would you handle a dynamic list of months without hardcoding them?
  3. JSON parsing pipeline. Create a table with a string column containing JSON payloads (use INSERT INTO with raw JSON strings). Write a query that parses the JSON, extracts 3 nested fields, and writes the result to a new Delta table. Verify schema enforcement by intentionally sending a malformed JSON string and observe the behavior.
  4. Explode and re-aggregate. Take a table with an items array column (create it with ARRAY(STRUCT(...))). EXPLODE the array to get one row per item. Then aggregate back to the order level calculating total item count and order total. Verify your totals match what you'd expect.
