Chapter 11

Unity Catalog

The three-level namespace, data governance, grants, row-level security, and lineage. Unity Catalog is the biggest platform change in Databricks in years.

Account, Workspace, and Metastore

Three concepts to keep straight: a Databricks account is your top-level organizational entity (one per company, billing lives here). A workspace is a collaborative environment within an account (you might have prod-workspace, staging-workspace, dev-workspace). A Unity Catalog metastore is a single governance layer that can be attached to multiple workspaces in the same account — so one set of catalogs, schemas, and grants applies across all workspaces. You typically have one metastore per region. Unity Catalog replaces the legacy per-workspace Hive metastore.

The Three-Level Namespace

Before Unity Catalog, Databricks used a two-level namespace: schema.table (or database.table). Unity Catalog adds a top-level catalog:

-- Legacy (Hive metastore): two levels
SELECT * FROM my_database.orders;

-- Unity Catalog: three levels
SELECT * FROM catalog_name.schema_name.table_name;

-- Set a default catalog so you don't have to type it
USE CATALOG my_catalog;
USE SCHEMA bronze;

-- Now these are equivalent
SELECT * FROM orders;
SELECT * FROM bronze.orders;
SELECT * FROM my_catalog.bronze.orders;

A typical catalog structure for a data platform:

prod_catalog/
  bronze/ ← raw ingested data
    orders_raw, customers_raw, events_raw
  silver/ ← cleaned, validated
    orders, customers, products
  gold/ ← business-ready aggregations
    revenue_daily, customer_ltv, funnel_metrics
  monitoring/ ← pipeline health
    pipeline_runs, data_quality_results

Grants and Permissions

-- Grant access to a schema
GRANT USE SCHEMA ON SCHEMA prod_catalog.gold TO `data-analysts@company.com`;
GRANT SELECT ON SCHEMA prod_catalog.gold TO `data-analysts@company.com`;

-- Grant access to a specific table only
GRANT SELECT ON TABLE prod_catalog.gold.revenue_daily TO `analyst-team`;

-- Grant write access (for ETL service accounts)
GRANT MODIFY ON SCHEMA prod_catalog.silver TO `etl-service-account`;
GRANT CREATE ON SCHEMA prod_catalog.silver TO `etl-service-account`;

-- Grant at the catalog level (all schemas)
GRANT USE CATALOG ON CATALOG prod_catalog TO `data-engineers`;

-- View existing grants
SHOW GRANTS ON TABLE prod_catalog.silver.orders;

-- Revoke
REVOKE SELECT ON TABLE prod_catalog.bronze.orders FROM `analyst-team`;
If You Know Basic PySpark

With the legacy Hive metastore, access control was typically handled at the cluster level (attach to a cluster, you can access everything). Unity Catalog shifts to fine-grained, table-level access control enforced centrally. The same permissions apply whether you access data through a notebook, a SQL warehouse, or a job. This is the main reason enterprises migrate to Unity Catalog.

Row-Level Security with Row Filters

Row filters restrict which rows a user can see without changing the query. They're applied transparently whenever the table is queried:

-- IS_ACCOUNT_GROUP_MEMBER() is a Unity Catalog built-in function that returns TRUE
-- if the current user is a member of the specified account-level group.

-- Store user → region mappings in a lookup table
CREATE TABLE prod_catalog.security.user_regions (
    email  STRING,
    region STRING
);
INSERT INTO prod_catalog.security.user_regions VALUES
    ('alice@company.com', 'US'),
    ('bob@company.com',   'EU');

-- Create a function that filters based on the lookup table
CREATE FUNCTION prod_catalog.security.orders_region_filter(region_col STRING)
RETURN IF(
    IS_ACCOUNT_GROUP_MEMBER('global-data-team'),
    true,
    region_col = (
        SELECT region FROM prod_catalog.security.user_regions
        WHERE email = current_user()
    )
);

-- Apply the row filter to a table
ALTER TABLE prod_catalog.silver.orders
SET ROW FILTER prod_catalog.security.orders_region_filter ON (region);

-- Now US analysts only see US rows, EU analysts only see EU rows
-- Global-data-team members see everything
SELECT * FROM prod_catalog.silver.orders;
-- Automatically filtered based on the querying user

Column Masking

-- Create a masking function
CREATE FUNCTION prod_catalog.security.mask_email(email STRING)
RETURN IF(
    IS_ACCOUNT_GROUP_MEMBER('pii-access-team'),
    email,
    CONCAT(LEFT(email, 2), '****@****.com')
);

-- Apply column mask
ALTER TABLE prod_catalog.silver.customers
ALTER COLUMN email SET MASK prod_catalog.security.mask_email;

-- PII team sees: john.doe@company.com
-- Others see:    jo****@****.com

External Locations

External locations are the Unity Catalog replacement for DBFS mounts. They give named access to S3 paths with IAM-controlled permissions:

-- Create a storage credential (once per AWS account, done by admin)
CREATE STORAGE CREDENTIAL prod_s3_credential
WITH IAM_ROLE 'arn:aws:iam::123456789:role/databricks-prod-role';

-- Create an external location pointing to S3
CREATE EXTERNAL LOCATION prod_data
URL 's3://my-prod-bucket/data/'
WITH (STORAGE CREDENTIAL prod_s3_credential);

-- Create a table at an external location
CREATE TABLE prod_catalog.bronze.orders_raw
LOCATION 's3://my-prod-bucket/data/orders/raw/'
USING DELTA;

-- Or create a Volume for unstructured files
CREATE VOLUME prod_catalog.bronze.raw_files
LOCATION 's3://my-prod-bucket/data/raw-files/';
# Access files via Volume path (no mounts needed)
df = spark.read.json("/Volumes/prod_catalog/bronze/raw_files/orders/2024/01/")

# Write to a Volume
df.write.format("json").save("/Volumes/prod_catalog/bronze/raw_files/processed/")

Data Lineage

Unity Catalog automatically tracks lineage: which tables were read/written by which queries and jobs. Access via the Catalog Explorer or programmatically:

# Lineage is available in the Catalog Explorer UI
# Programmatic access via the Databricks SDK
from databricks.sdk import WorkspaceClient

w = WorkspaceClient()

# Get table lineage (upstream sources)
lineage = w.data_lineage.get_table_lineage(
    table_name="prod_catalog.gold.revenue_daily"
)
for upstream in lineage.upstreams:
    print(f"Source: {upstream.table_info.full_name}")
Lineage Limitations

Lineage is tracked for SQL queries run on Databricks compute attached to Unity Catalog. It doesn't track operations done via direct S3 access outside Databricks, JDBC connections, or older workspaces not on Unity Catalog. Column-level lineage is supported for direct column references but not computed expressions.

Exercises
  1. Namespace migration. If you have any tables in the legacy Hive metastore (database.table), write a migration plan to move them to Unity Catalog. What's the SYNC command? What needs to change in your notebook paths?
  2. Design a permission model. Your team has: data engineers (read/write all layers), analysts (read gold only), external partners (read one specific gold table). Write the GRANT statements to implement this in Unity Catalog.
  3. Implement a row filter. Create a table with a country column. Write a row filter function that returns all rows if the user is in a 'global-access' group, otherwise only rows matching a hardcoded test country. Verify it works by querying as different users (or groups).
