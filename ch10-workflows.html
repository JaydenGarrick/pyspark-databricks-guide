Chapter 10

Workflows & Orchestration

Databricks Jobs, task DAGs, cluster types, parameters, task values, and error handling. How to turn a notebook into a production pipeline.

Jobs vs Interactive Notebooks

Interactive notebooks are for development. Databricks Jobs are for production. The key differences:

AspectInteractive NotebookDatabricks Job
Cluster typeAll-purpose (expensive)Job cluster (30-50% cheaper DBUs)
SchedulingManualCron, event-triggered, or manual
ParametersWidgets set manuallyPassed as job parameters
Error handlingManual retryAutomatic retry with configurable policy
AlertingNoneEmail/webhook on failure
Audit trailNotebook revision historyRun history with logs per run

Task Types

A Databricks Job is a DAG of Tasks. Each task can be one of several types:

  • Notebook task — runs a Databricks notebook; parameters passed as widget values
  • Python script task — runs a .py file from a Git repo or DBFS; parameters as command-line args or env vars
  • Python wheel task — runs a packaged Python wheel with an entry point
  • SQL task — runs SQL against a SQL warehouse (not a cluster); good for queries and dashboards
  • dbt task — runs a dbt project natively
  • Spark JAR task — runs a compiled Scala/Java JAR
Notebook vs Python Script for Production

Notebooks are convenient but have limitations: they can't be unit-tested with standard Python tools, they're harder to code-review in Git (JSON format with output), and they mix code with display output. For serious production pipelines, consider migrating notebook logic to Python wheel packages. Databricks Asset Bundles (DABs) is Databricks' infrastructure-as-code framework: you define jobs, clusters, permissions, and notebooks in YAML files in your Git repo and deploy them with the databricks bundle deploy CLI command. DABs make this workflow manageable — keep notebooks for exploration, define production job configs in code.

Task Dependencies

Tasks within a job form a DAG. You define dependencies in the job UI or JSON definition:

{
  "name": "daily-etl-pipeline",
  "tasks": [
    {
      "task_key": "ingest_bronze",
      "notebook_task": { "notebook_path": "/pipelines/bronze/ingest_orders" },
      "job_cluster_key": "etl_cluster"
    },
    {
      "task_key": "process_silver",
      "depends_on": [{"task_key": "ingest_bronze"}],
      "notebook_task": { "notebook_path": "/pipelines/silver/process_orders" },
      "job_cluster_key": "etl_cluster"
    },
    {
      "task_key": "build_gold_revenue",
      "depends_on": [{"task_key": "process_silver"}],
      "notebook_task": { "notebook_path": "/pipelines/gold/revenue_metrics" },
      "job_cluster_key": "etl_cluster"
    },
    {
      "task_key": "build_gold_customers",
      "depends_on": [{"task_key": "process_silver"}],
      "notebook_task": { "notebook_path": "/pipelines/gold/customer_metrics" },
      "job_cluster_key": "etl_cluster"
    }
  ]
}

In this example, build_gold_revenue and build_gold_customers both depend on process_silver but not on each other — they run in parallel after silver completes.

Job Cluster vs All-Purpose Cluster

{
  "job_clusters": [
    {
      "job_cluster_key": "etl_cluster",
      "new_cluster": {
        "spark_version": "14.3.x-scala2.12",
        "node_type_id": "r5.2xlarge",
        "num_workers": 4,
        "aws_attributes": {
          "availability": "SPOT_WITH_FALLBACK"
        },
        "spark_conf": {
          "spark.sql.adaptive.enabled": "true",
          "spark.databricks.delta.optimizeWrite.enabled": "true"
        },
        "auto_termination_minutes": 0  // Job clusters terminate when the job finishes — not via idle timeout
      }
    }
  ]
}

To reuse an existing all-purpose cluster (for debugging only, not production):

{
  "tasks": [{
    "task_key": "debug_task",
    "existing_cluster_id": "1234-567890-abc12345"
  }]
}

Job Parameters and Task Values

Parameters pass values into notebooks as widget values or Python script args. Task values pass outputs from one task to another:

# In the upstream task (bronze ingestion)
# After processing, record how many rows were ingested
rows_ingested = df_bronze.count()

# Pass this value to downstream tasks
dbutils.jobs.taskValues.set(key="rows_ingested", value=rows_ingested)
dbutils.jobs.taskValues.set(key="run_date", value=str(date.today()))
dbutils.jobs.taskValues.set(key="has_data", value=rows_ingested > 0)
# In a downstream task
rows_ingested = dbutils.jobs.taskValues.get(
    taskKey="ingest_bronze",
    key="rows_ingested",
    default=0
)
run_date = dbutils.jobs.taskValues.get(
    taskKey="ingest_bronze",
    key="run_date"
)

print(f"Processing {rows_ingested} rows for {run_date}")

# Conditional logic based on upstream result
if rows_ingested == 0:
    print("No data to process — skipping silver transform")
    dbutils.notebook.exit("SKIPPED")

Error Handling and Retries

{
  "tasks": [{
    "task_key": "process_silver",
    "retry_on_timeout": true,
    "max_retries": 3,
    "min_retry_interval_millis": 60000,
    "timeout_seconds": 3600
  }]
}

For task-level error handling within a notebook:

import traceback

try:
    # Main processing logic
    df = spark.table("catalog.bronze.orders")
    processed = transform_silver(df)
    write_silver(processed)
    dbutils.jobs.taskValues.set("status", "SUCCESS")
    dbutils.jobs.taskValues.set("rows_written", processed.count())

except Exception as e:
    error_msg = f"Silver processing failed: {str(e)}\n{traceback.format_exc()}"
    print(error_msg)

    # Log to an error table
    spark.createDataFrame([(str(date.today()), "silver_orders", error_msg)],
                           ["run_date", "pipeline", "error"]) \
        .write.format("delta").mode("append") \
        .saveAsTable("catalog.monitoring.pipeline_errors")

    dbutils.jobs.taskValues.set("status", "FAILED")
    raise  # re-raise to mark the task as failed

Quartz Cron vs Standard Unix Cron

Databricks uses Quartz cron format, which has 6 fields (seconds minutes hours day-of-month month day-of-week), not the 5-field Unix cron you may know. The ? wildcard means "no specific value" and is used when you've specified day-of-month OR day-of-week but not both (they conflict). If you're used to 0 6 * * * in Unix cron, the Quartz equivalent is 0 0 6 * * ?. The Databricks Jobs UI includes a cron builder that generates the correct format for you.

Common cron expressions:

  • 0 0 6 * * ? — 6 AM every day
  • 0 0 */4 * * ? — every 4 hours
  • 0 0 8 ? * MON-FRI — 8 AM weekdays only
  • 0 0/30 * * * ? — every 30 minutes
Exercises
  1. Convert a notebook to a job. Take a notebook you've been running manually. Create a Databricks Job with a job cluster (not your interactive cluster). Add two widgets for run_date and env. Run the job manually once, inspect the run output and logs.
  2. Build a two-task DAG. Create a job with two tasks: task A generates a summary count, stores it with dbutils.jobs.taskValues.set, and task B reads it with dbutils.jobs.taskValues.get and prints it. Verify the value passes correctly.
  3. Add retry logic. Intentionally break a task (divide by zero, bad table name). Configure 3 retries with 60-second backoff. Run the job and observe the retry behavior in the run timeline. Then fix the bug and confirm the job succeeds.
