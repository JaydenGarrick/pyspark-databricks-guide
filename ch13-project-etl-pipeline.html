Project: ETL Pipeline

Build a production-grade Bronze → Silver → Gold medallion ETL pipeline using Delta Lake, MERGE INTO, and schema enforcement.

What We're Building

You're building a daily order processing pipeline for an e-commerce platform. Raw JSON event data lands in S3 every hour. Your pipeline needs to ingest it into Bronze (raw), clean and validate it to Silver (trusted), and aggregate it to Gold (analytics-ready). The pipeline must handle late-arriving data, duplicate events, and schema drift gracefully.

Starter & Finished Projects

Companion files live in projects/etl-pipeline/ in this repository:

  • starter/pipeline.py — Schemas and sample data pre-defined; TODO 1–TODO 9 for you to implement
  • finished/pipeline.py — Complete working solution with numbered inline comments

To use in Databricks: import this repo via Workspace → Repos → Add Repo, then open the file in a notebook. Run cells top-to-bottom. Each cell is separated by # COMMAND ---------- (Databricks notebook cell delimiter).

Pipeline Architecture

The medallion architecture separates concerns across three Delta tables:

LayerTableWhat it holdsOperations
Bronzeorders_rawExact copy of source JSON; append-only; no transformationsAPPEND with dedup on event_id
SilverordersValidated, cleaned, typed; one row per orderMERGE (upsert on order_id)
Goldrevenue_dailyAggregated daily metrics per product/regionMERGE with overwrite for the day's window
If You Know Basic PySpark

You've probably written ETL as: read → transform → overwrite. The medallion pattern replaces that overwrite with MERGE. This makes the pipeline idempotent: running it twice on the same input produces the same result. It also enables easy reprocessing: drop the Bronze rows for a bad date range and re-run — Silver and Gold self-correct on the next run.

Why Deduplicate at Bronze?

Source systems (Kafka, SQS, webhooks) deliver at-least-once. The same event can arrive multiple times. Deduplicating at Bronze on event_id means Silver and Gold don't need to worry about event-level duplicates. Bronze dedup handles message delivery retries; Silver's MERGE handles order-level updates (an order moving from "pending" to "completed" is the same order, not a duplicate).

from delta.tables import DeltaTable

# Read and transform bronze
bronze_df = spark.table("pipeline_demo.bronze.orders_raw")

silver_df = bronze_df \
    .dropDuplicates(["order_id", "event_ts"]) \
    .withColumn("quantity",   col("quantity").cast(IntegerType())) \
    .withColumn("unit_price", col("unit_price").cast(DecimalType(10, 2))) \
    .withColumn("amount",     (col("quantity") * col("unit_price")).cast(DecimalType(10, 2))) \
    .withColumn("order_date", to_date(col("event_ts").cast(TimestampType()))) \
    .withColumn("ingested_at", current_timestamp()) \
    .filter(col("unit_price").isNotNull() & col("quantity").isNotNull()) \
    .select(
        "order_id", "customer_id", "product_id",
        "quantity", "unit_price", "amount",
        "status", "region", "order_date", "ingested_at"
    )

silver_table = DeltaTable.forName(spark, "pipeline_demo.silver.orders")
silver_table.alias("target").merge(
    silver_df.alias("source"),
    "target.order_id = source.order_id"
).whenMatchedUpdate(set={
    "status":      "source.status",
    "ingested_at": "source.ingested_at"
}).whenNotMatchedInsertAll().execute()

Extension Challenges
  1. Add Change Data Feed. Enable CDF on the silver table. After running the pipeline twice (with one order status changing from "pending" to "completed"), query table_changes() to see only the changed rows. This is how you'd feed a downstream notification service.
  2. Parameterize by date. Add a run_date widget to the notebook. Modify Bronze ingestion to filter event_ts to only the specified date. Modify Gold to overwrite only that date's partition. This makes the pipeline safe for backfill.
  3. Add a quarantine table. Instead of filtering null-price rows out silently in Silver, write them to a pipeline_demo.monitoring.quarantine table with a rejection_reason column. This is prerequisite knowledge for Chapter 15's data quality project.
