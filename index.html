PySpark to Databricks

A deep dive for engineers who already know the basics. You can read a CSV and do a groupBy — now let's make you dangerous.

Why This Guide If You Already Know PySpark?

You can create a SparkSession, read a DataFrame, filter rows, and write to Parquet. That's a solid foundation. But there's a chasm between "I can run PySpark" and "I understand what Databricks is actually doing and why my jobs cost $400/run."

This guide targets the gap. It assumes basic PySpark fluency and teaches the concepts that actually matter for production Databricks work: Delta Lake internals, Spark SQL beyond SELECT *, join strategies, the Catalyst optimizer, Unity Catalog, and AWS integration.

What You Already Know
  • SparkSession, DataFrames, RDDs
  • df.filter(), df.groupBy(), df.join()
  • Reading CSV/Parquet from a path
  • Writing output with df.write.parquet()
  • Basic Spark SQL: SELECT, WHERE, GROUP BY
  • Column expressions with F.col(), F.lit()
What This Guide Adds
  • Delta Lake — ACID transactions, time travel, MERGE
  • Spark SQL mastery — window functions, CTEs, QUALIFY
  • Performance reasoning — partitioning, Z-order, broadcast vs sort-merge
  • Catalyst & AQE — reading query plans, optimizer behavior
  • Databricks platform — Workflows, Unity Catalog, secrets
  • AWS integration — S3 access patterns, instance profiles
How "If You Know Basic PySpark" Callouts Work

Throughout this guide, amber callout boxes like this one bridge what you already know in PySpark to the Databricks-native concept being introduced. If something feels new, check these boxes first — they give you the anchor before the deep dive.

The Mental Shift

Basic PySpark teaches you what to type. This guide teaches you why it works and when to choose differently. The difference between a PySpark user and a Databricks engineer is understanding the execution layer: what Catalyst does with your query, why that join spilled to disk, and why your Delta table is 50,000 small files.

Spark SQL as Your Anchor

Most PySpark developers underuse Spark SQL. They reach for the DataFrame API because it feels more "programmatic," but SQL is often clearer, faster to write, and equally performant. This guide treats Spark SQL as a first-class citizen — Part II is dedicated entirely to it, including features like QUALIFY, window frames, and lateral column aliases that most engineers don't know exist.

Databricks Runtime Assumed

This guide targets Databricks Runtime 13.3 LTS or later (which ships Spark 3.4+). Some features — Liquid Clustering, EXECUTE IMMEDIATE, Unity Catalog — require specific DBR versions and are noted inline. If you're on an older runtime, the core concepts still apply but specific APIs may differ.

Real Talk: What This Guide Is Not

This is not an intro to PySpark, a Databricks certification study guide, or a definitive reference for every API. It is a practitioner-focused deep dive that teaches the concepts experienced Databricks engineers use daily. Some topics (streaming, MLflow, Databricks SQL warehouse tuning) are mentioned but not covered in depth — they each deserve their own guide.

How to Use This Guide

Every chapter is structured the same way:

  1. Basic PySpark anchor first — we start from what you already know
  2. Side-by-side comparisons — PySpark DataFrame API vs SQL vs Databricks-native
  3. Honest trade-offs — every feature has a cost, and we discuss it
  4. Quizzes — test your understanding before moving on
  5. Exercises — hands-on extension challenges

If you have solid basic PySpark, Parts I and II will feel partially familiar and partially revelatory. Delta Lake (Chapter 3) and window functions (Chapter 4) are where most engineers level up. Part III is where you stop guessing why jobs are slow and start knowing.

Prerequisites

You should be comfortable with: creating a SparkSession, reading/writing DataFrames, basic column expressions (F.col, F.lit, F.when), groupBy and agg, join, and basic SQL (SELECT, WHERE, GROUP BY, JOIN). No Databricks-specific knowledge is assumed.
