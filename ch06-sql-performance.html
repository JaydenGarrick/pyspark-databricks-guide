Chapter 6

SQL Performance Patterns

Reading query plans, join hints, predicate pushdown, and statistics. Stop guessing why your SQL is slow.

EXPLAIN: Reading Query Plans

Before optimizing anything, read the plan. EXPLAIN shows what Spark intends to do with your query. Start with EXPLAIN FORMATTED — it's the most readable format for day-to-day debugging. Use EXPLAIN EXTENDED when you want to compare what you wrote to what Catalyst rewrote. Use EXPLAIN COST when you want to see Spark's row-count and size estimates for each operator (useful for diagnosing why the optimizer made a bad join decision).

-- Show the optimized logical plan (most useful for SQL)
EXPLAIN FORMATTED SELECT * FROM orders WHERE customer_id = 'C001';

-- Show all plan stages: parsed → analyzed → optimized → physical
EXPLAIN EXTENDED SELECT * FROM orders o JOIN customers c USING (customer_id);

-- Show only the physical plan with stage/cost info
EXPLAIN COST SELECT * FROM orders GROUP BY customer_id;
# PySpark equivalent
df = spark.sql("SELECT * FROM orders WHERE customer_id = 'C001'")
df.explain("formatted")   # same as EXPLAIN FORMATTED
df.explain("extended")    # same as EXPLAIN EXTENDED
df.explain("cost")        # same as EXPLAIN COST

Key things to look for in a physical plan:

Node TypeWhat it meansPerformance signal
FileScanReading files from storageCheck PushedFilters — missing filters = full scan
BroadcastHashJoinSmall table broadcast to all workersGood — no shuffle needed
SortMergeJoinBoth sides shuffled and sortedExpected for large-large joins, check for skew
HashAggregateIn-memory hash aggregationLook for partial + final stages (two-phase)
ExchangeShuffle between stagesCount shuffles — each is expensive I/O
SortSorting within a stageNeeded for merge joins and ORDER BY

Annotated example of a physical plan for SELECT * FROM orders WHERE customer_id = 'C001':

== Physical Plan ==
*(1) Project [order_id#10, customer_id#11, amount#12]     <-- Project: column selection
+- *(1) Filter (isnotnull(customer_id#11)                 <-- Filter: WHERE clause
        AND (customer_id#11 = C001))
   +- *(1) FileScan delta default.orders                  <-- FileScan: reading files
           [order_id#10, customer_id#11, amount#12]
           Batched: true, DataFilters: [isnotnull(...)]
           PushedFilters: [IsNotNull(customer_id),        <-- Good: pushed to storage
                           EqualTo(customer_id,C001)]
           ReadSchema: struct<order_id:string,...>

Predicate Pushdown

Predicate pushdown means filters are applied at the file/storage level, not after reading all data into memory. Look for PushedFilters in the FileScan node:

-- This WILL push the filter to the file scan
-- (Delta min/max stats allow skipping files where customer_id not in range)
SELECT * FROM catalog.silver.orders
WHERE customer_id = 'C001'
  AND event_date >= '2024-01-01';

-- Check the plan:
-- FileScan delta [...] PushedFilters: [IsNotNull(customer_id), EqualTo(customer_id,C001), ...]
-- This may NOT push down correctly if you wrap the column in a function
-- Spark can't push down filters on transformed columns
SELECT * FROM catalog.silver.orders
WHERE LOWER(customer_id) = 'c001';  -- function wrapping prevents pushdown
-- Fix: store customer_id already lowercased, or filter on stored case
Functions Kill Pushdown

Wrapping a column in any function — LOWER(), CAST(), DATE_TRUNC(), YEAR() — usually prevents predicate pushdown for that column. The exception is when the function can be pushed to the storage format layer. For date filtering, filter on the raw timestamp or a partition column, not on DATE_TRUNC() of a timestamp column.

Join Hints

Sometimes Spark's optimizer chooses the wrong join strategy. Hints let you override it:

-- Force a broadcast join (small table on the right)
SELECT /*+ BROADCAST(c) */ *
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id;

-- Alternative syntax
SELECT /*+ BROADCASTJOIN(customers) */ *
FROM orders
JOIN customers USING (customer_id);

-- Force sort-merge join
SELECT /*+ MERGE(o, c) */ *
FROM orders o
JOIN customers c USING (customer_id);

-- Force shuffle hash join (faster than sort-merge for non-sorted data)
SELECT /*+ SHUFFLE_HASH(o) */ *
FROM orders o
JOIN customers c USING (customer_id);

-- Force shuffle-and-replicate (use when one side is very small)
SELECT /*+ SHUFFLE_REPLICATE_NL(c) */ *
FROM orders o
JOIN customers c ON o.category = c.preferred_category;
# PySpark hint API
df = orders.join(
    customers.hint("broadcast"),
    on="customer_id"
)

# Or use the SQL hint in spark.sql()
result = spark.sql("""
    SELECT /*+ BROADCAST(c) */ o.*, c.email
    FROM orders o
    JOIN customers c USING (customer_id)
""")
When to Use Broadcast Hints

Broadcast hints are appropriate when one side is genuinely small (<200MB after filtering, not before). Databricks auto-broadcasts tables below spark.sql.autoBroadcastJoinThreshold (default 10MB). If your dimension table is 50MB but not auto-broadcasting, check whether Spark has current table statistics — it may not know the size. After ANALYZE TABLE, auto-broadcast may kick in without needing a hint.

Table Statistics and CBO

Spark's cost-based optimizer (CBO) uses table statistics to choose join order and join strategies. Without statistics, it guesses — and often wrong:

-- Compute and store table-level statistics
ANALYZE TABLE catalog.silver.orders COMPUTE STATISTICS;

-- Compute column-level statistics (needed for join ordering)
ANALYZE TABLE catalog.silver.orders
COMPUTE STATISTICS FOR COLUMNS customer_id, event_date, amount;

-- All columns
ANALYZE TABLE catalog.silver.orders COMPUTE STATISTICS FOR ALL COLUMNS;

-- View stored statistics
DESCRIBE EXTENDED catalog.silver.orders;
# Check if statistics are stale (key diagnostic step)
spark.sql("DESCRIBE EXTENDED catalog.silver.orders") \
    .filter("col_name IN ('Statistics', 'numFiles', 'sizeInBytes')") \
    .display()
Delta Statistics Are Automatic for File-Level

Delta Lake automatically maintains min/max statistics per column for each Parquet file as part of the transaction log. These are used for data skipping (skipping entire files that can't contain matching rows). This is different from CBO statistics. ANALYZE TABLE adds row-count and column-cardinality statistics at the table level, which helps the optimizer choose join order and estimate output sizes.

Caching and Persistence

-- Cache a table in memory (SQL)
CACHE TABLE catalog.silver.customers;

-- Cache with lazy evaluation (doesn't trigger a scan until first use)
CACHE LAZY TABLE catalog.silver.customers;

-- Remove from cache
UNCACHE TABLE catalog.silver.customers;

-- Check what's cached
-- sys.caches is a Databricks-specific system view; returns one row per cached table/DataFrame
-- with columns: name, data_type (SPARK vs DELTA), sizeInBytes, numPartitions, isLazy
SELECT * FROM sys.caches;
# Cache a DataFrame
df_customers = spark.table("catalog.silver.customers").cache()
df_customers.count()  # triggers the cache load

# Persist with a specific storage level
from pyspark.storagelevel import StorageLevel
df_customers.persist(StorageLevel.MEMORY_AND_DISK)

# Release cache
df_customers.unpersist()
Caching vs Delta Caching

Databricks has two distinct caches. (1) Spark cache (df.cache(), CACHE TABLE): caches deserialized data in executor memory/disk. Lost when cluster is restarted. (2) Delta cache (automatic on I3/I3en instances): I3 and I3en are AWS EC2 instance families with local NVMe SSD storage. Databricks uses that local SSD to cache raw Parquet file data, so subsequent reads hit local disk instead of S3. This cache persists across queries and notebook sessions. The Delta cache is usually better for large tables you query repeatedly. The Spark cache is better for DataFrames you reference multiple times in a single job.

Common SQL Anti-Patterns

Anti-PatternProblemFix
SELECT * on wide tablesReads all columns; misses projection pruningSelect only needed columns
DISTINCT on large result setsFull shuffle + sort for dedupUse ROW_NUMBER() QUALIFY if dedup key is known
Functions on filter columnsPrevents predicate pushdownFilter on raw column value or a precomputed column
Multiple COUNT(DISTINCT)Each is a separate shuffle passUse approx_count_distinct() or combine into one pass with COLLECT_SET
Correlated subqueriesRe-executed once per outer rowRewrite as JOIN or window function
Large IN() lists (>100 values)Serialization overhead and poor planningUse a JOIN to a temp table or broadcast variable
Exercises
  1. Read a query plan. Write a query that joins two large tables. Run EXPLAIN FORMATTED and identify: what join strategy was chosen? Were filters pushed down? How many shuffles occur? Now add a BROADCAST hint for the smaller table and compare the plans.
  2. Statistics impact. Run ANALYZE TABLE on a table you query frequently. Before and after, run the same query and compare the query plans. Did join ordering or strategy change? Measure the execution time difference.
  3. Find a pushdown failure. Write a query where you filter using a function on a column (e.g., YEAR(event_date) = 2024). Check the plan for PushedFilters. Then rewrite using a range filter (event_date BETWEEN '2024-01-01' AND '2024-12-31') and compare. Which has better pushdown?
