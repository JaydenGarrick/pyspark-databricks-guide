Chapter 4

Window Functions & Analytics

The most powerful and underused feature in Spark SQL. Every analytics query you've been solving with self-joins, you should have been solving with window functions.

What Is a Window Function?

A window function computes a value for each row relative to a set of rows defined by a window. Unlike GROUP BY which collapses rows into a single aggregate, window functions keep all rows and add a computed column alongside them.

The canonical example: "for each order, what is the customer's cumulative spend up to this order?"

  • With GROUP BY: impossible in one query — GROUP BY collapses all of a customer's orders into one row.
  • With a self-join: possible but expensive — join every order to all prior orders for the same customer.
  • With a window function: one pass, no join, elegant SQL.
If You Know Basic PySpark

PySpark has window functions too — from pyspark.sql import Window. You may have written w = Window.partitionBy("customer_id").orderBy("created_at") and df.withColumn("rn", F.row_number().over(w)). This chapter covers the same concepts but shows the SQL syntax, which is often cleaner, plus Databricks-specific extensions like QUALIFY that don't exist in the PySpark API.

The OVER() Clause

Every window function uses an OVER() clause that defines:

  • PARTITION BY — divides rows into groups (like GROUP BY, but doesn't collapse)
  • ORDER BY — ordering within each partition (required for ranking, lag/lead, and cumulative aggregates)
  • ROWS/RANGE BETWEEN — the window frame (which rows around the current row to include)
SELECT
    order_id,
    customer_id,
    amount,
    created_at,
    -- The function goes BEFORE the OVER()
    SUM(amount) OVER (
        PARTITION BY customer_id      -- one window per customer
        ORDER BY created_at           -- ordered chronologically
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW  -- running total
    ) AS cumulative_spend
FROM orders;

Ranking Functions

ROW_NUMBER

Assigns a unique sequential integer to each row within the partition. The most common use: deduplication.

SQL
SELECT *
FROM (
  SELECT
    *,
    ROW_NUMBER() OVER (
      PARTITION BY customer_id
      ORDER BY created_at DESC
    ) AS rn
  FROM orders
)
WHERE rn = 1;
-- Gets most recent order per customer
PySpark API
from pyspark.sql import Window
import pyspark.sql.functions as F
# F is the standard alias for pyspark.sql.functions — the module with col(), lit(),
# row_number(), sum(), when(), and hundreds of other built-in functions.
# You'll see `F.col()`, `F.sum()`, etc. throughout PySpark code.

w = Window \
  .partitionBy("customer_id") \
  .orderBy(F.col("created_at").desc())

df \
  .withColumn("rn", F.row_number().over(w)) \
  .filter(F.col("rn") == 1) \
  .drop("rn")

RANK vs DENSE_RANK vs ROW_NUMBER

These three functions look similar but handle ties differently:

SELECT
    player,
    score,
    ROW_NUMBER() OVER (ORDER BY score DESC) AS row_num,
    RANK()       OVER (ORDER BY score DESC) AS rank,
    DENSE_RANK() OVER (ORDER BY score DESC) AS dense_rank
FROM leaderboard;

  • ROW_NUMBER: Always unique. Ties are broken arbitrarily. Use for deduplication.
  • RANK: Ties get the same rank, then there's a gap (1,1,3,4). Use for "top N" where you want gaps.
  • DENSE_RANK: Ties get the same rank, no gaps (1,1,2,3). Use for "top N" where you want no gaps.

NTILE

Divides rows into N roughly equal buckets:

SELECT
    customer_id,
    total_spent,
    NTILE(4) OVER (ORDER BY total_spent DESC) AS spend_quartile
FROM customer_summary;
-- quartile 1 = top 25%, quartile 4 = bottom 25%

LAG and LEAD

LAG and LEAD access values from other rows in the partition without a self-join. They're essential for period-over-period calculations:

SELECT
    event_date,
    daily_revenue,
    -- Previous day's revenue
    LAG(daily_revenue, 1, 0) OVER (ORDER BY event_date) AS prev_day_revenue,
    -- Next day's revenue
    LEAD(daily_revenue, 1, 0) OVER (ORDER BY event_date) AS next_day_revenue,
    -- Day-over-day change
    daily_revenue - LAG(daily_revenue, 1, 0) OVER (ORDER BY event_date) AS dod_change,
    -- % change (NULL safe)
    ROUND(
        (daily_revenue - LAG(daily_revenue) OVER (ORDER BY event_date))
        / NULLIF(LAG(daily_revenue) OVER (ORDER BY event_date), 0) * 100,
        2
    ) AS dod_pct_change
FROM daily_revenue;

LAG/LEAD signature: LAG(expr, offset, default)

  • expr — the column to look back at
  • offset — how many rows to look back (default 1)
  • default — value when there's no prior row (default NULL)
Week-over-Week with LAG

For weekly comparisons, use LAG(metric, 7) if you have daily data, or partition by week number. For monthly comparisons in a table that has daily rows, partition by DATE_TRUNC('month', event_date) and use LAG(metric, 1) within those partitions. Match your granularity to your LAG offset.

Aggregate Window Functions

Any aggregate function (SUM, AVG, COUNT, MIN, MAX) can be used with OVER():

SELECT
    order_id,
    customer_id,
    amount,
    -- Running total per customer
    SUM(amount) OVER (
        PARTITION BY customer_id
        ORDER BY created_at
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS running_total,

    -- Rolling 7-day average
    AVG(amount) OVER (
        PARTITION BY customer_id
        ORDER BY created_at
        ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
    ) AS rolling_7day_avg,

    -- Total for the entire partition (no ORDER BY = entire partition)
    SUM(amount) OVER (PARTITION BY customer_id) AS customer_total,

    -- % of customer's total this order represents
    ROUND(amount / SUM(amount) OVER (PARTITION BY customer_id) * 100, 2) AS pct_of_total,

    -- Running count
    COUNT(*) OVER (
        PARTITION BY customer_id
        ORDER BY created_at
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS order_number
FROM orders;

Window Frames: ROWS vs RANGE

The frame clause defines which rows to include relative to the current row. This is the most nuanced part of window functions:

-- ROWS frame: physical rows relative to current
ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW   -- all prior rows through current
ROWS BETWEEN 2 PRECEDING AND 2 FOLLOWING           -- 2 rows before through 2 after
ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING  -- entire partition

-- RANGE frame: logical rows with same ORDER BY value (handles ties)
RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW  -- includes rows with same value
RANGE BETWEEN INTERVAL '7' DAYS PRECEDING AND CURRENT ROW  -- 7-day range window
ROWS vs RANGE: Tie Handling

When you use ORDER BY without a frame clause, Spark defaults to RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. For SUM, this means tied rows all get the same running total (all ties are summed together). If you want strict per-row running totals, use ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. For most use cases, ROWS is what you actually want.

QUALIFY: Filtering on Window Results

QUALIFY is a Databricks/Spark SQL clause (Spark 3.3+) that filters on the result of a window function — without needing a subquery. It's one of the most useful extensions over standard SQL and dramatically simplifies deduplication queries:

Without QUALIFY (standard SQL)
-- Need a subquery to filter on window result
SELECT * FROM (
  SELECT
    *,
    ROW_NUMBER() OVER (
      PARTITION BY customer_id
      ORDER BY created_at DESC
    ) AS rn
  FROM orders
)
WHERE rn = 1;
With QUALIFY (Databricks/Spark SQL)
-- QUALIFY filters on the window result directly
-- No subquery needed, no aliasing
SELECT *
FROM orders
QUALIFY ROW_NUMBER() OVER (
  PARTITION BY customer_id
  ORDER BY created_at DESC
) = 1;

QUALIFY is especially powerful for multi-condition deduplication:

-- Get top 3 orders per customer by amount, but only for active customers
SELECT
    customer_id,
    order_id,
    amount,
    status
FROM orders
JOIN customers USING (customer_id)
WHERE customers.is_active = true
  AND orders.status != 'cancelled'
QUALIFY RANK() OVER (
    PARTITION BY orders.customer_id
    ORDER BY amount DESC
) <= 3;
QUALIFY Execution Order

SQL clauses execute in this order: FROM → WHERE → GROUP BY → HAVING → window functions → QUALIFY → SELECT → ORDER BY → LIMIT. This means QUALIFY can reference window functions computed in the SELECT list, and it applies after WHERE (pre-aggregation filtering). Think of QUALIFY as "HAVING, but for window functions."

FIRST_VALUE and LAST_VALUE

FIRST_VALUE and LAST_VALUE return the first or last value in the window frame. FIRST_VALUE is straightforward. LAST_VALUE has a critical gotcha: by default, the frame ends at the current row, so it always returns the current row's value — not the last row in the partition.

SELECT
    customer_id,
    order_id,
    amount,
    created_at,
    -- FIRST_VALUE: works as expected — always the first order amount per customer
    FIRST_VALUE(amount) OVER (
        PARTITION BY customer_id
        ORDER BY created_at
    ) AS first_order_amount,

    -- LAST_VALUE with DEFAULT frame: returns current row (wrong!)
    -- LAST_VALUE(amount) OVER (PARTITION BY customer_id ORDER BY created_at)

    -- LAST_VALUE with explicit UNBOUNDED frame: returns last row in partition (correct)
    LAST_VALUE(amount) OVER (
        PARTITION BY customer_id
        ORDER BY created_at
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS last_order_amount
FROM orders;

When you use LAST_VALUE, always specify ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING to extend the frame to the full partition. Otherwise you get the current row's value — which is usually not what you want.

Named Windows

If you use the same window definition multiple times, define it once with a WINDOW clause:

SELECT
    order_id,
    customer_id,
    amount,
    ROW_NUMBER()  OVER w AS rn,
    RANK()        OVER w AS rank,
    SUM(amount)   OVER w AS running_total,
    AVG(amount)   OVER w AS running_avg
FROM orders
WINDOW w AS (
    PARTITION BY customer_id
    ORDER BY created_at
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
);

Named windows can also be extended:

SELECT
    order_id,
    customer_id,
    amount,
    SUM(amount) OVER cust_win         AS customer_running_total,
    SUM(amount) OVER (cust_win ORDER BY created_at) AS ordered_running_total
FROM orders
WINDOW cust_win AS (PARTITION BY customer_id);

Practical Patterns

Cohort Analysis

-- Assign each customer to an acquisition cohort (first order month)
-- then calculate retention by cohort
WITH cohorts AS (
    SELECT
        customer_id,
        DATE_TRUNC('month', MIN(created_at)) AS cohort_month
    FROM orders
    GROUP BY customer_id
),
customer_activity AS (
    SELECT
        o.customer_id,
        c.cohort_month,
        DATE_TRUNC('month', o.created_at) AS activity_month,
        -- MONTHS_BETWEEN returns a decimal; FLOOR converts to whole months
        -- Note: Spark SQL's DATEDIFF only works in days; use MONTHS_BETWEEN for month-level diffs
        CAST(FLOOR(MONTHS_BETWEEN(
            DATE_TRUNC('month', o.created_at),
            c.cohort_month
        )) AS INT) AS months_since_acquisition
    FROM orders o
    JOIN cohorts c USING (customer_id)
)
SELECT
    cohort_month,
    months_since_acquisition,
    COUNT(DISTINCT customer_id) AS active_customers,
    ROUND(
        COUNT(DISTINCT customer_id) * 100.0
        / FIRST_VALUE(COUNT(DISTINCT customer_id)) OVER (
            PARTITION BY cohort_month
            ORDER BY months_since_acquisition
        ),
        1
    ) AS retention_rate_pct
FROM customer_activity
GROUP BY cohort_month, months_since_acquisition
ORDER BY cohort_month, months_since_acquisition;

Keep Latest Row Per Key

-- Deduplicate a CDC stream: keep only the most recent record per ID
-- This is one of the most common operations in Silver-layer processing
SELECT * FROM raw_events
QUALIFY ROW_NUMBER() OVER (
    PARTITION BY entity_id
    ORDER BY event_timestamp DESC, sequence_num DESC
) = 1;

Percentiles and Histograms

SELECT
    product_category,
    sale_amount,
    PERCENTILE_CONT(0.25) WITHIN GROUP (ORDER BY sale_amount)
        OVER (PARTITION BY product_category) AS p25,
    PERCENTILE_CONT(0.50) WITHIN GROUP (ORDER BY sale_amount)
        OVER (PARTITION BY product_category) AS median,
    PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY sale_amount)
        OVER (PARTITION BY product_category) AS p75,
    NTILE(100) OVER (
        PARTITION BY product_category
        ORDER BY sale_amount
    ) AS percentile_rank
FROM sales;
Quick Check

You want to keep only the first purchase per customer (by date), and filter out any results where the first purchase amount was less than $10. Which approach is most efficient in Databricks?

  • Self-join orders to itself on customer_id, filter WHERE o1.created_at = MIN(o2.created_at), then filter amount > 10
  • Use GROUP BY customer_id MIN(created_at), then join back to the original table, then filter amount > 10
  • Use QUALIFY ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY created_at) = 1, then WHERE amount > 10 in the outer query or add AND amount > 10 after QUALIFY
  • Use DISTINCT ON (not supported in Spark SQL)
QUALIFY with ROW_NUMBER. QUALIFY ROW_NUMBER() OVER (...) = 1 is the most efficient and readable approach. The WHERE filter can be applied either before (reduces rows before windowing) or after (filters window results). For this case: add WHERE amount >= 10 before the QUALIFY to push the filter earlier, then QUALIFY handles the deduplication: WHERE amount >= 10 QUALIFY ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY created_at) = 1.
Exercises
  1. Running total with reset. Given a table of daily sales by region, write a query that calculates a running total that resets at the start of each month. Hint: partition by both region and DATE_TRUNC('month', sale_date).
  2. First and last value. Use FIRST_VALUE() and LAST_VALUE() to get, for each order, the customer's first-ever order amount and their most recent order amount (before the current one). Why does LAST_VALUE require a specific frame clause to work correctly?
  3. QUALIFY deduplication. Take a table with intentional duplicates (same primary key, multiple rows). Use QUALIFY to deduplicate, keeping the row with the highest version number or most recent timestamp. Verify by counting rows before and after.
  4. Gap detection. Given a table of user sessions with start/end times, use LAG to identify sessions where there was more than 30 minutes of inactivity since the previous session. These represent "new visit" boundaries for session stitching.
  5. Build the cohort chart. Using the cohort analysis query from this chapter as a starting point, run it against a real or sample orders table. Pivot the result so cohort months are rows and months-since-acquisition are columns (0, 1, 2, ..., 12). Hint: use the PIVOT clause from Chapter 5.
