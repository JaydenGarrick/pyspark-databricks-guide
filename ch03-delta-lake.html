Chapter 3

Delta Lake

Parquet, but with a transaction log. ACID guarantees, time travel, upserts, and schema evolution on top of object storage.

The Problem with Parquet

Parquet is a great columnar storage format. But it's just a file format — it has no concept of transactions, schema enforcement, or atomicity. This causes real problems:

  • Partial writes: If a job fails halfway through writing, you get a corrupt dataset with some new files and some old files mixed together.
  • No upserts: To update records in Parquet, you rewrite the entire table. There's no way to update one row without reading and rewriting everything.
  • Schema drift: Nothing stops a job from writing a DataFrame with a different schema. Readers silently get null columns or cast errors.
  • Read-write conflicts: Concurrent writers produce undefined behavior. One writer can corrupt another's output.
If You Know Basic PySpark

You may have used df.write.mode("overwrite").parquet("/path") extensively. Delta Lake replaces this pattern. Instead of managing files yourself, you manage tables. Delta handles atomicity, versioning, and optimization automatically. The trade-off is a metadata overhead (~1-2% of table size for the transaction log) and some added complexity for advanced features.

Delta Lake Architecture: The Transaction Log

Delta Lake adds a _delta_log/ directory alongside your Parquet files. This directory contains JSON transaction log files (one per commit) that record every operation on the table:

my_table/
  _delta_log/
    00000000000000000000.json ← initial create
    00000000000000000001.json ← first insert
    00000000000000000002.json ← second insert
    00000000000000000010.checkpoint.parquet ← checkpoint every 10 commits
    00000000000000000011.json
  part-00000-abc.snappy.parquet
  part-00001-def.snappy.parquet

Each JSON log entry records which files were added and removed, the schema, statistics for each file (min/max values per column), and the operation type. This log is what enables ACID guarantees, time travel, and optimistic concurrency control.

Creating Delta Tables

Basic PySpark (Parquet)
# Write
df.write \
  .mode("overwrite") \
  .parquet("s3://bucket/my-table/")

# Read
df = spark.read \
  .parquet("s3://bucket/my-table/")
Delta Lake
# Write (as managed table in Unity Catalog)
df.write \
  .format("delta") \
  .mode("overwrite") \
  .saveAsTable("catalog.schema.my_table")

# Or as external table at a path
df.write \
  .format("delta") \
  .mode("overwrite") \
  .save("s3://bucket/my-table/")

# Read a Delta table
df = spark.read.format("delta") \
  .load("s3://bucket/my-table/")
# or by name
df = spark.table("catalog.schema.my_table")
Managed vs External Tables

This is a critical distinction. Managed tables (saveAsTable() or CREATE TABLE without LOCATION) store data in a Databricks-managed location (your default catalog storage). When you DROP TABLE, Databricks deletes the underlying files. External tables (.save("s3://...") or CREATE TABLE ... LOCATION 's3://...') store data at a path you control. Dropping the table only removes the metadata — the files remain. For production data you care about, prefer external tables (or Unity Catalog managed tables in a dedicated catalog with UC storage credentials) so a DROP TABLE doesn't wipe your data.

Using SQL is often cleaner for table creation:

-- Create an empty Delta table with explicit schema
CREATE TABLE catalog.schema.orders (
    order_id    STRING NOT NULL,
    customer_id STRING,
    amount      DECIMAL(10,2),
    status      STRING,
    created_at  TIMESTAMP
)
USING DELTA
COMMENT 'Raw orders from the e-commerce platform'
TBLPROPERTIES (
    -- See Chapter 7 for what these do and when to enable them
    'delta.autoOptimize.optimizeWrite' = 'true',
    'delta.autoOptimize.autoCompact' = 'true'
);

-- Create Delta table from a SELECT
CREATE TABLE catalog.schema.orders_us
USING DELTA
AS SELECT * FROM catalog.schema.orders WHERE country = 'US';

Time Travel

Every write to a Delta table creates a new version. You can read any historical version:

from delta.tables import DeltaTable  # needed for all DeltaTable operations

# View the table's history
dt = DeltaTable.forName(spark, "catalog.schema.orders")
dt.history().show(truncate=False)

# Read a specific version (version number from history)
df_v5 = spark.read.format("delta") \
    .option("versionAsOf", 5) \
    .table("catalog.schema.orders")

# Read as of a specific timestamp
df_yesterday = spark.read.format("delta") \
    .option("timestampAsOf", "2024-01-14 00:00:00") \
    .table("catalog.schema.orders")

In SQL:

-- Read version 5
SELECT * FROM catalog.schema.orders VERSION AS OF 5;

-- Read as of a timestamp
SELECT * FROM catalog.schema.orders
TIMESTAMP AS OF '2024-01-14T00:00:00.000Z';

-- Describe the history
DESCRIBE HISTORY catalog.schema.orders;
Time Travel Retention

Historical versions are retained until you run VACUUM. The default retention period is 7 days (delta.deletedFileRetentionDuration = 'interval 7 days'). Increase this for audit requirements, but know that longer retention means more storage cost. You cannot time-travel beyond the retention window — the old Parquet files have been deleted by VACUUM.

MERGE INTO — Upserts and CDC

MERGE INTO is the single most powerful Delta Lake operation. It enables CDC (Change Data Capture) patterns, SCD Type 1/2, and incremental loads without full table rewrites:

from delta.tables import DeltaTable  # already imported above — shown here for clarity

target = DeltaTable.forName(spark, "catalog.silver.customers")

# Incoming changes (new or updated records)
updates_df = spark.table("catalog.bronze.customer_updates")

target.alias("tgt").merge(
    updates_df.alias("src"),
    condition="tgt.customer_id = src.customer_id"
).whenMatchedUpdate(set={
    "email":      "src.email",
    "name":       "src.name",
    "updated_at": "src.updated_at"
}).whenNotMatchedInsert(values={
    "customer_id": "src.customer_id",
    "email":       "src.email",
    "name":        "src.name",
    "created_at":  "src.created_at",
    "updated_at":  "src.updated_at"
}).execute()

The SQL equivalent is often more readable:

MERGE INTO catalog.silver.customers AS tgt
USING catalog.bronze.customer_updates AS src
ON tgt.customer_id = src.customer_id

WHEN MATCHED THEN UPDATE SET
    tgt.email      = src.email,
    tgt.name       = src.name,
    tgt.updated_at = src.updated_at

WHEN NOT MATCHED THEN INSERT (
    customer_id, email, name, created_at, updated_at
) VALUES (
    src.customer_id, src.email, src.name, src.created_at, src.updated_at
)

-- Optional: delete records marked as deleted in the source
WHEN MATCHED AND src.is_deleted = true THEN DELETE;
MERGE Performance Considerations

MERGE is powerful but expensive. It reads the target table to find matching rows, which can be a full scan without proper partitioning or Z-ordering. For large tables, always ensure the merge key column is either a partition column or Z-ordered. A poorly designed MERGE on a 1TB table can take hours. See Chapter 7 (Data Layout) for how to optimize for MERGE workloads.

OPTIMIZE and VACUUM

Delta tables accumulate small files over time from frequent writes (especially streaming or incremental loads). OPTIMIZE compacts them:

-- Compact small files (default target: 1GB per file)
OPTIMIZE catalog.silver.orders;

-- Compact a specific partition only
OPTIMIZE catalog.silver.orders
WHERE event_date = '2024-01-15';

-- Compact AND apply Z-Order clustering on a column
OPTIMIZE catalog.silver.orders
ZORDER BY (customer_id);
# Python equivalent
from delta.tables import DeltaTable

dt = DeltaTable.forName(spark, "catalog.silver.orders")
dt.optimize().executeCompaction()

# With Z-Order
dt.optimize().executeZOrderBy("customer_id")

After OPTIMIZE, old small files are no longer active but still exist on disk until VACUUM removes them:

-- Remove files older than the retention period (default 7 days)
VACUUM catalog.silver.orders;

-- Override retention (minimum 0 hours — dangerous, breaks time travel)
-- The 168-hour check can be disabled, but don't do this in production
VACUUM catalog.silver.orders RETAIN 168 HOURS;
VACUUM and Time Travel

Running VACUUM permanently deletes historical Parquet files. After VACUUM, you cannot time-travel to versions before the retention window. If your team relies on time travel for auditing or debugging, be very deliberate about when and how often you VACUUM. A common pattern: VACUUM weekly with 30-day retention for production audit tables.

Schema Evolution and Enforcement

By default, Delta enforces schema: writing a DataFrame with an unexpected column fails. You can relax this with schema evolution:

# This fails if df has columns not in the table schema (default behavior)
df.write.format("delta").mode("append").saveAsTable("catalog.silver.orders")

# Enable schema evolution: adds new columns automatically
df.write \
    .format("delta") \
    .mode("append") \
    .option("mergeSchema", "true") \
    .saveAsTable("catalog.silver.orders")

# For MERGE operations, enable schema evolution at the session level
spark.conf.set("spark.databricks.delta.schema.autoMerge.enabled", "true")

Or enable it as a table property:

ALTER TABLE catalog.silver.orders
SET TBLPROPERTIES ('delta.columnMapping.mode' = 'name');

-- Add a column without rewriting the table
ALTER TABLE catalog.silver.orders
ADD COLUMN loyalty_tier STRING AFTER customer_id;

-- Rename a column (requires column mapping enabled)
ALTER TABLE catalog.silver.orders
RENAME COLUMN loyalty_tier TO customer_tier;

Change Data Feed (CDF)

CDF lets downstream consumers read only the rows that changed since their last read — inserts, updates, and deletes — without scanning the full table:

-- Enable CDF on a table
ALTER TABLE catalog.silver.orders
SET TBLPROPERTIES ('delta.enableChangeDataFeed' = 'true');

-- Read all changes since version 10
SELECT * FROM table_changes('catalog.silver.orders', 10)
ORDER BY _commit_version, _commit_timestamp;
# Read changes since a specific version
changes = spark.read.format("delta") \
    .option("readChangeFeed", "true") \
    .option("startingVersion", 10) \
    .table("catalog.silver.orders")

# _change_type values:
#   'insert'           — new row added
#   'update_preimage'  — the row's state BEFORE an update (the old values)
#   'update_postimage' — the row's state AFTER an update (the new values)
#   'delete'           — row was deleted
# To get only the current state of updated rows, filter for 'update_postimage'
changes.filter("_change_type IN ('insert', 'update_postimage')").display()

Quick Check

You run VACUUM catalog.silver.orders RETAIN 0 HOURS. What happens to time travel?

  • Time travel still works for versions created today
  • All historical Parquet files are deleted; you can no longer time-travel to any prior version
  • VACUUM fails because the minimum retention is 24 hours
  • Time travel works but only for the current version
All historical files are deleted. RETAIN 0 HOURS removes all files not referenced by the current table version. You lose all time travel capability for this table. Databricks normally prevents this with a safety check (spark.databricks.delta.retentionDurationCheck.enabled), but if overridden, it's permanent. The minimum enforced by the safety check is 168 hours (7 days).
Exercises
  1. Create a Delta table from scratch. Using SQL, create a Delta table with at least 5 columns. Insert 100+ rows using INSERT INTO ... VALUES or a CTAS (CREATE TABLE ... AS SELECT). Run DESCRIBE HISTORY and DESCRIBE DETAIL — understand every field in the output.
  2. Implement a MERGE pattern. Create a "source" DataFrame with 50 rows: 30 that already exist in your target table (updates) and 20 that are new (inserts). Write and execute a MERGE statement. Verify the result count and that updated rows reflect the new values.
  3. Break and fix a schema. Write a DataFrame with an extra column not in your Delta table's schema. Observe the error. Then re-run with mergeSchema=true. Inspect the updated schema with DESCRIBE TABLE.
  4. Time travel debugging. Write three batches of data to a Delta table (simulating three daily loads). Run DESCRIBE HISTORY to see three versions. Now read version 1 using time travel and confirm the row count matches your first batch only.
