Chapter 7

Data Layout & Partitioning

How you store data is as important as how you query it. Disk partitioning, Z-Order, Liquid Clustering, and file size — the decisions that determine your scan cost.

Why Data Layout Matters

Spark's performance on Delta tables is dominated by I/O. A query that touches 100GB of data and returns 1,000 rows is fundamentally limited by how much of that 100GB it can skip. Data layout is the set of decisions that determine how much you can skip.

Three mechanisms work together to skip data:

  1. Partition pruning — skip entire directories (partitions) based on partition column values
  2. File skipping — skip entire Parquet files based on Delta's min/max statistics
  3. Column pruning — read only the columns needed (columnar format benefit)

Disk Partitioning

Partitioning divides a Delta table into subdirectories based on column values. Queries that filter on partition columns skip entire directories without any I/O:

# Write with partitioning
df.write \
    .format("delta") \
    .partitionBy("event_date", "region") \
    .mode("overwrite") \
    .saveAsTable("catalog.silver.events")
# Stores as: event_date=2024-01-01/region=US/part-0001.snappy.parquet
-- CREATE TABLE with partition columns
CREATE TABLE catalog.silver.events (
    event_id    STRING,
    customer_id STRING,
    event_type  STRING,
    amount      DECIMAL(10,2),
    event_date  DATE,
    region      STRING
)
USING DELTA
PARTITIONED BY (event_date, region);

-- Query: only reads event_date=2024-01-15/region=US/ directory
SELECT * FROM catalog.silver.events
WHERE event_date = '2024-01-15'
  AND region = 'US';
If You Know Basic PySpark

You may have used .partitionBy() with Parquet. Delta partitioning works the same way structurally, but Delta also maintains transaction log statistics that enable file-level skipping within a partition. This means even within a single partition directory, Delta can skip files that don't contain matching rows based on min/max values — something raw Parquet cannot do.

Partitioning Guidelines

GuidelineReason

Don't over-partition (avoid compound partitions on 3+ columns)Compound partitions multiply the partition count rapidly
The Small File Problem

Every streaming write or incremental INSERT creates new Parquet files. Over time, a table accumulates thousands of tiny files (the "small file problem"). Tiny files are expensive: each file requires a separate S3 LIST and GET request, and Spark creates one task per file. Run OPTIMIZE regularly (at least daily for frequently written tables) to compact small files into larger ones.

Z-Order Clustering

Z-Order co-locates related data within files based on multiple columns. Unlike partitioning (which creates separate directories), Z-Order works at the file level using a space-filling curve — a mathematical technique that maps multi-dimensional coordinates (multiple column values) into a single sequence while preserving locality. Rows with similar values in any of the Z-ordered columns end up physically close together in the same files. This lets Delta's min/max statistics efficiently skip files even for multi-column filters, because related data is not scattered randomly across thousands of files.

-- Apply Z-Order when running OPTIMIZE
OPTIMIZE catalog.silver.orders
ZORDER BY (customer_id, event_date);

-- For a specific partition
OPTIMIZE catalog.silver.orders
WHERE event_date >= '2024-01-01'
ZORDER BY (customer_id);

Z-Order is powerful for columns that are: (1) frequently used in WHERE clauses, (2) high-cardinality (many distinct values), (3) not partition columns.

Without Z-Order
-- 1 year of order data, partitioned by date
-- Query for customer C001's orders
SELECT * FROM orders
WHERE event_date BETWEEN '2023-01-01' AND '2024-01-01'
  AND customer_id = 'C001'

-- Must scan all files across 365 partitions
-- C001's data is scattered randomly across files
-- Files skipped: 0 (no Z-Order stats for customer_id)
With Z-Order on customer_id
-- After: OPTIMIZE orders ZORDER BY (customer_id)
SELECT * FROM orders
WHERE event_date BETWEEN '2023-01-01' AND '2024-01-01'
  AND customer_id = 'C001'

-- C001's data is co-located in a few files per partition
-- Delta's min/max stats for customer_id allow skipping
-- most files — typically 80-95% of files skipped
Z-Order Limitations

Effectiveness degrades beyond 4 columns: Z-Order on 5+ columns provides diminishing returns because the space-filling curve becomes too complex. Stick to 2-3 high-impact columns. Not cumulative: Z-Order must be re-applied after new data is written. Newly written files are not Z-ordered until the next OPTIMIZE. Full file rewrite: OPTIMIZE with ZORDER rewrites all affected files, which is expensive for large tables. Consider ZORDER only on columns that appear in most point-lookup queries.

Liquid Clustering (DBR 13.3+)

Liquid Clustering is the modern replacement for both partitioning and Z-Order in Databricks. It clusters data incrementally without requiring full table rewrites, and adapts automatically as data changes:

-- Create a table with Liquid Clustering
CREATE TABLE catalog.silver.orders (
    order_id    STRING,
    customer_id STRING,
    event_date  DATE,
    amount      DECIMAL(10,2)
)
CLUSTER BY (customer_id, event_date);

-- Change clustering keys without rewriting the table
ALTER TABLE catalog.silver.orders
CLUSTER BY (event_date, region);  -- just changes future clustering behavior
-- Trigger clustering (like OPTIMIZE, but incremental)
OPTIMIZE catalog.silver.orders;  -- automatically applies Liquid Clustering for Liquid tables

Recommendation: Use Liquid Clustering for New Tables

For new Delta tables on DBR 13.3+, default to Liquid Clustering instead of partitioning + Z-Order. It handles high-cardinality clustering keys (like customer_id) that would be catastrophic as partition columns, adapts to changing access patterns without full rewrites, and requires less tuning expertise. The main caveat: it's not supported by all Spark clients outside Databricks yet.

File Size Tuning

-- Check current file sizes and counts
-- DESCRIBE HISTORY returns an 'operationMetrics' column of type MAP
-- Access individual metrics with dot notation or map subscript ['key']
SELECT
    operation,
    operationMetrics.numOutputRows AS rows_written,
    operationMetrics.numFiles      AS files_written,
    operationMetrics.numOutputBytes / 1024 / 1024 AS mb_written
FROM (DESCRIBE HISTORY catalog.silver.orders)
ORDER BY timestamp DESC
LIMIT 10;

-- Set target file size for OPTIMIZE (default 1GB)
ALTER TABLE catalog.silver.orders
SET TBLPROPERTIES ('delta.targetFileSize' = '134217728');  -- 128MB

-- Enable Auto Optimize (write-time compaction)
ALTER TABLE catalog.silver.orders
SET TBLPROPERTIES (
    'delta.autoOptimize.optimizeWrite' = 'true',  -- better bin-packing on write
    'delta.autoOptimize.autoCompact'   = 'true'   -- async compaction after write
);
Auto Optimize vs Manual OPTIMIZE

optimizeWrite reorganizes task output before writing to produce better-sized files (prevents tiny files from forming). autoCompact triggers a lightweight compaction job after writes complete. Both are recommended for tables with frequent incremental writes. They don't replace periodic full OPTIMIZE — they just reduce how often you need to run it.

Exercises
  1. Partition analysis. Run DESCRIBE DETAIL catalog.silver.your_table on a table you own. Look at numFiles and sizeInBytes. Calculate average file size. Is it in the 128MB-1GB sweet spot? If not, what would you change — partition strategy, OPTIMIZE frequency, or Auto Optimize settings?
  2. Z-Order impact measurement. Take a Delta table with at least 100 files. Run a query that filters on a non-partition column. Record the number of files scanned (visible in the Spark UI job details or EXPLAIN output). Run OPTIMIZE ... ZORDER BY on that column. Re-run the same query and compare files scanned.
  3. Partition design exercise. You're building a Silver table for clickstream events: ~1 billion rows/day, 50GB/day, queried by user_id, event_date, and event_type. Design the partitioning strategy. Which column(s) should be partition columns? Which should be Z-ordered? How large will each partition be?
