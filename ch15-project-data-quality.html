Chapter 15

Project: Data Quality Framework

Build a reusable, data-driven quality check framework with a quarantine table, quality score tracking, and a Delta-backed monitoring report.

What We're Building

Data pipelines fail silently. A null price slips through, a duplicated order_id inflates revenue, a future date gets ingested from a misconfigured source. You need a framework that catches these proactively — not a one-off assert, but a reusable system that: (1) runs configurable checks, (2) quarantines bad rows without crashing the pipeline, and (3) writes check results to a monitoring table so quality trends are visible over time.

Starter & Finished Projects

Companion files live in projects/data-quality/:

  • starter/data_quality.py — DataQualityCheck class skeleton with TODO 1–TODO 8
  • finished/data_quality.py — Complete framework with all checks implemented

Framework Architecture

Three components working together:

ComponentResponsibility
DataQualityCheckDefines a single check: a name, a SQL expression that returns True/False per row, and a severity (WARNING vs ERROR)
DataQualityRunnerRuns a list of checks against a DataFrame; routes passing rows to the target table and failing rows to quarantine
monitoring.dq_resultsDelta table that accumulates check run history: run_date, table, check_name, pass_count, fail_count, fail_pct, severity
If You Know Basic PySpark

You've probably written quality checks as: assert df.filter(col("amount").isNull()).count() == 0. That crashes the pipeline. This framework replaces the assert with a quarantine write — bad rows go to a side table with a reason column, the pipeline continues with clean rows, and you get trend data to see if quality is improving or degrading over time.

Step 1: Define Quality Checks as Data

Catalog Setup

This project writes to pipeline_demo.monitoring.dq_results. If you completed Chapter 13, the catalog and schemas already exist. If starting fresh, run the setup cell from Chapter 13 (Step 0) first. The monitoring schema may not exist yet: spark.sql("CREATE SCHEMA IF NOT EXISTS pipeline_demo.monitoring").

from functools import reduce  # needed for combining check failure flags
from dataclasses import dataclass
from typing import List, Optional
from pyspark.sql import DataFrame
from pyspark.sql.functions import *  # includes expr(), col(), lit(), current_timestamp()

@dataclass
class DataQualityCheck:
    name: str
    expression: str       # SQL expression returning boolean — True = PASS
    severity: str = "ERROR"  # "ERROR" (quarantine) or "WARNING" (log only)
    description: str = ""

# Define checks for the orders table
ORDER_CHECKS = [
    DataQualityCheck(
        name="amount_not_null",
        expression="amount IS NOT NULL",
        severity="ERROR",
        description="Order amount cannot be null"
    ),
    DataQualityCheck(
        name="amount_positive",
        expression="amount > 0",
        severity="ERROR",
        description="Order amount must be positive"
    ),
    DataQualityCheck(
        name="valid_status",
        expression="status IN ('pending', 'completed', 'shipped', 'cancelled')",
        severity="ERROR",
        description="Status must be a known value"
    ),
    DataQualityCheck(
        name="order_date_not_future",
        expression="order_date <= current_date()",
        severity="WARNING",
        description="Order date should not be in the future"
    ),
    DataQualityCheck(
        name="customer_id_not_null",
        expression="customer_id IS NOT NULL",
        severity="ERROR",
        description="Customer ID cannot be null"
    ),
]

Monitoring Dashboard Query

-- Quality score trend over time
SELECT
    run_timestamp,
    table_name,
    SUM(pass_count) AS total_passed,
    SUM(fail_count) AS total_failed,
    ROUND(SUM(pass_count) * 100.0 / SUM(pass_count + fail_count), 1) AS overall_pass_pct
FROM pipeline_demo.monitoring.dq_results
GROUP BY run_timestamp, table_name
ORDER BY run_timestamp;

-- Which checks fail most often?
SELECT
    check_name,
    severity,
    COUNT(*) AS run_count,
    ROUND(AVG(fail_pct), 2) AS avg_fail_pct,
    MAX(fail_pct) AS max_fail_pct
FROM pipeline_demo.monitoring.dq_results
GROUP BY check_name, severity
ORDER BY avg_fail_pct DESC;
Databricks Native: Delta Live Tables Expectations

Delta Live Tables (DLT) has built-in data quality expectations via @dlt.expect() decorators that do exactly what this project implements manually. If you're using DLT, use native expectations instead. This project teaches the underlying mechanics so you understand what DLT is doing — and so you can build quality checks in non-DLT pipelines.

Extension Challenges
  1. Threshold-based alerting. Add a threshold_pct field to DataQualityCheck. If fail_pct > threshold_pct for an ERROR check, raise a DataQualityException that fails the pipeline task. Add retry-with-investigation logic: if the check failed due to a known transient issue (null coming from an upstream system), the quarantine + retry pattern should self-heal on the next run.
  2. Schema drift detection. Add a schema_check that compares the incoming DataFrame's schema to an expected schema (stored as JSON in a DBFS volume). If any expected column is missing or has a different type, flag it as a schema drift ERROR. This catches upstream schema changes before they corrupt Silver.
  3. Quality SLA report. Write a query against monitoring.dq_results that computes a 7-day rolling average fail percentage per check. Flag any check where the rolling average exceeds 5% (your SLA). This is a real-world monitoring dashboard query — use the window functions from Chapter 4.
